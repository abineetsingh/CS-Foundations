Transactions & ACID

One skill, learned today: wrap multi-step changes so they happen entirely or not at all — and name what can still go wrong when two of them race.

Why this matters

Every system that touches money, inventory, or ownership lives or dies on this lesson. Interviewers probe it with one word — "what if it crashes halfway?" — and production incidents probe it without asking. ACID is also the baseline that phase 3's distributed systems will painfully renegotiate.

The problem: multi-step truth

Transfer $50 from Ada to Grace = two writes: subtract from Ada, add to Grace. Crash between them and $50 has vanished — the database is internally consistent-looking but the books are wrong. A transaction brackets the steps into one all-or-nothing unit:

BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE name = 'ada';
UPDATE accounts SET balance = balance + 50 WHERE name = 'grace';
COMMIT;                -- both happen…
-- or ROLLBACK;        -- …or neither does

ACID, word by word

LetterPromiseConcretely
Atomicityall or nothingcrash mid-transaction → automatic rollback; no half-transfers ever visible
Consistencyrules hold before and afterconstraints (lesson 12) like balance ≥ 0 can't be violated by a committed transaction
Isolationconcurrent transactions don't corrupt each otherthe subtle one — see below
Durabilitycommitted = survives crashthe DB appends to a write-ahead log and fsyncs before saying "committed" [SQLite: WAL]

Isolation: where the bodies are buried

Two transactions running "at once" are actually interleaved. The classic corruption is the lost update: T1 and T2 both read Ada's balance (100), both compute 100−50, both write 50 — one withdrawal evaporated. Databases offer isolation levels, a dial trading correctness against concurrency [DDIA ch. 8]:

LevelGuards againstNotes
Read committedseeing others' uncommitted writescommon default (Postgres); lost updates still possible!
Repeatable read / snapshotdata shifting under you mid-transactioneach transaction sees a frozen snapshot
Serializableeverything — as if run one-by-onethe gold standard; costs throughput, may abort+retry

The working engineer's toolkit for lost updates, worth reciting: atomic single statements (SET balance = balance - 50 — let the DB do the read-modify-write), SELECT … FOR UPDATE (lock the row first), or serializable + retry. Never read-in-app-then-write for contested values.

Hands-on: watch atomicity work (5 min)

sqlite3 bank.db
CREATE TABLE accounts (name TEXT PRIMARY KEY, balance INTEGER CHECK (balance >= 0));
INSERT INTO accounts VALUES ('ada', 100), ('grace', 100);

BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE name = 'ada';
SELECT * FROM accounts;      -- ada 50 (inside the transaction's view)
ROLLBACK;                    -- simulate a crash / abort
SELECT * FROM accounts;      -- ada 100 — the subtraction never happened

BEGIN;
UPDATE accounts SET balance = balance - 200 WHERE name = 'ada';   -- CHECK fails!
-- constraint violation; ROLLBACK; nothing changed. C in action.
AI-lab tie-in

Agent tool-use wants transactions and mostly can't have them: "book the flight, then charge the card" has no BEGIN/COMMIT spanning two external APIs. That's the distributed-transaction problem (lesson 27 — sagas, compensation), and it's why idempotency (02, 22) obsesses agent-infrastructure engineers. Knowing what single-database ACID gives you is how you notice when you've left it.
