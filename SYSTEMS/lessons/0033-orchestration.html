Orchestration: What Kubernetes Actually Does

One skill, learned today: explain the four jobs of an orchestrator — and the declare-and-reconcile idea that powers them — without drowning in Kubernetes vocabulary.

Why this matters

Kubernetes runs a huge share of the world's backends and every serious ML platform; "K8s" fluency is assumed in infra conversations at AI labs. But interviews test the concepts, not YAML trivia — and the core concept (reconciliation loops) is one of the most transferable ideas in all of systems… including agent design.

The gap it fills

You can run one container (0032). Production runs thousands, across hundreds of machines. Someone must decide which machine runs what, restart the dead, scale the busy, and route traffic to wherever things landed. Doing this by hand is a full-time job for a large team; an orchestrator (Kubernetes, ECS, Nomad) is that team, automated [K8s overview]. Four jobs:

The big idea: declare, then reconcile

You do not tell Kubernetes what to do. You declare a desired state — "12 replicas of notes-api:v2, each 2 CPU" — into its store (etcd — yes, the consensus store from 0026, guarding the cluster's one precious truth). Then reconciliation loops run forever: observe actual state → diff against desired → act to close the gap. A machine dying isn't an emergency requiring a runbook; it's just a new diff (observed 9, desired 12 → start 3). Failure handling stops being an event and becomes the steady state of the system [K8s: controllers]:

Minimal vocabulary that makes K8s docs readable — deliberately just four words: a Pod is the unit of running (one or more containers scheduled together); a Deployment declares "N replicas of this pod spec" (and handles rolling updates — 0034); a Service is the stable name + load balancing across a pod set; a Node is a machine in the fleet. Everything else is elaboration.

AI-lab tie-in

GPU scheduling is the orchestration boss level — the scheduler must pack jobs onto scarce H100s, gang-schedule (a 512-GPU training job needs all 512 simultaneously), and preempt low-priority work for launches. And notice the deeper rhyme: observe → diff → act toward a declared goal is also a decent description of an agent loop. Reconciliation thinking is the transferable asset — Kubernetes is just its most industrialized deployment.
