Linked Lists II: Reversal & Fast/Slow Pointers

Today's skill: the two techniques behind nearly every linked-list interview question — in-place reversal and the tortoise-and-hare.

Why this, now

"Reverse a Linked List" is a filter question: interviewers use it to separate people who can juggle three pointers from people who memorized an answer. Fast/slow is subtler — a genuinely clever idea you'll enjoy owning.

Technique 1: in-place reversal

Flip every arrow. Walk the list carrying two pointers — prev (done part) and current (remaining part) — and repoint one arrow per step:

def reverse_list(head):
    prev = None
    current = head
    while current:
        nxt = current.next     # 1. save the rest before breaking the link
        current.next = prev    # 2. flip this arrow backward
        prev = current         # 3. advance prev
        current = nxt          # 4. advance current
    return prev                # prev ends on the old tail = new head

None ← 3 ← 1 ← 4 after three iterations of 3 → 1 → 4. O(n) time, O(1) space. The four lines must happen in exactly this order — line 1 is the "don't lose the chain" rule from Lesson 12. Write this function from memory until it's automatic; it reappears inside harder problems (reverse in k-groups, palindrome list).

Technique 2: fast & slow pointers

Two pointers, same start, different speeds — slow moves one hop, fast moves two.

Find the middle (one pass)

def middle(head):
    slow = fast = head
    while fast and fast.next:
        slow = slow.next           # 1 hop
        fast = fast.next.next      # 2 hops
    return slow                    # fast hit the end → slow is halfway

When fast has gone the whole way, slow has gone half. No length precomputation, no second pass.

Detect a cycle (Floyd's algorithm)

def has_cycle(head):
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:           # identity, not equality of values
            return True
    return False

The argument, worth saying aloud: if there's a cycle, both pointers eventually orbit inside it; fast gains one position per step, so the gap shrinks to zero — they must meet. No cycle → fast reaches None. O(n) time, O(1) space (the O(n)-space alternative — a visited set — is worth mentioning, then beating).

Composition: is this list a palindrome?

Middle (fast/slow) + reverse the second half + walk both halves comparing. O(n) time, O(1) space — three lessons of tools snapped together. Try it before looking anything up.

Mission link · ML/AI

Cycle detection matters wherever structures self-reference: Python's garbage collector hunts reference cycles, and computation-graph builders must reject cycles (a neural net's forward pass must be a DAG — Lesson 30 makes this precise). Floyd's trick also detects cycles in any iterated function — including pseudo-random generators.
