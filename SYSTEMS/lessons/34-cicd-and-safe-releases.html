CI/CD & Safe Releases

One skill, learned today: ship changes to a live system the way production teams do — pipeline, canary, instant rollback — and treat deploys as the risk they statistically are.

Why this matters

Most outages aren't caused by hardware or traffic — they're caused by changes we made. How you release is therefore a reliability decision, not a tooling detail. Interviewers probe it with "how do you deploy this safely?", and the capstone will ship through a pipeline you understand.

CI: the pipeline that guards the artifact

Continuous Integration: every commit automatically triggers a pipeline — lint → tests → build the container image (32) → push to registry. The output is a verified, immutable, versioned artifact (notes-api:git-abc123). No artifact ships that didn't pass; no human builds "special" versions on laptops. CD (continuous delivery/deployment) extends the pipeline to environments: staging automatically, production via strategy below [GitHub Actions].

Release strategies: exposure control

All modern strategies exploit the same property: immutable v1 still exists while v2 rolls out (32's insight). The game is limiting how many users can be hurt by a bad version, and how fast you can un-hurt them:

StrategyHowTradeoff
Rollingreplace pods a few at a time (K8s Deployments' default — 33)simple, no extra capacity; both versions serve at once (compat needed!); blast radius grows as it proceeds
Blue/greenfull v2 fleet stands up beside v1; flip the router; keep v1 warminstant flip and instant rollback; costs 2× capacity during the change
Canaryv2 gets 1% of traffic → watch error rate & latency (35's metrics!) → 5% → 25% → 100%; auto-rollback on regressionthe production standard: real traffic as the final test, blast radius capped at each stage

Two force-multipliers to name alongside: feature flags decouple deploy from release — code ships dark, the feature turns on per-user/percentage at runtime, and "rollback" of a feature is a toggle, not a deploy (this is how big launches actually happen). And compatibility discipline: rolling/canary means old and new run simultaneously, so DB migrations must be backward-compatible (expand → migrate → contract: add the new column, ship code reading both, remove the old later — never rename in one step).

AI-lab tie-in

Model releases are canary deployments with eval-shaped metrics: new model serves a traffic slice; dashboards compare refusal rates, latency, quality scores against baseline; regression → auto-rollback to the prior checkpoint (immutable artifacts — a model version is the ultimate one). And A/B testing is the canary mechanism pointed at product questions instead of safety ones. Same machinery, three uses.
