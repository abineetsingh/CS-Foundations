Partial Failure: Timeouts, Retries, Idempotency

One skill, learned today: handle the defining ambiguity of distributed systems — "no response" — without hanging forever or charging a customer twice.

Why this matters

Phase 3 begins with its founding fact: in a distributed system, things fail partially. A single machine either works or it doesn't; two machines connected by a network can each be fine while the conversation between them dies. Every pattern in this phase — queues, guarantees, sagas — descends from today's trio: timeout, retry, idempotency. Interviewers probe it with five words: "what if the call fails?"

The ambiguity at the bottom of everything

You call the payment service. No response. Which happened? (a) the request never arrived; (b) it arrived and the server crashed mid-work; (c) the work completed and the reply got lost. From your side these are indistinguishable — and (c) is the poisonous one: the customer was charged and you don't know it [DDIA ch. 9]. Everything below is machinery for acting sanely despite not knowing.

The trio, assembled

ToolRuleWhy
Timeoutevery network call has a deadline — no exceptions, everwithout one, a dead dependency's callers hang, their threads pile up, and the failure climbs the stack. Waiting forever is not patience; it's propagation
Retry (with exponential backoff + jitter)wait 1s, 2s, 4s… ± random, capped attemptsimmediate retries kick a struggling server while it's down — a self-inflicted stampede (10). Backoff gives it air; jitter stops synchronized re-attack [AWS Builders' Library]
Idempotency keyclient sends a unique id per operation; server stores processed ids and returns the saved result for repeatsmakes retries safe — the second "charge $50 [key=abc123]" is recognized and answered, not re-executed. This is how Stripe works [Stripe: idempotent requests]

Say the composed rule in interviews: "Timeouts bound the wait, backoff-with-jitter paces the retries, and idempotency keys make retrying safe — so 'no response' becomes a non-event." One more habit: retry budgets — retries multiply traffic exactly when the system is weakest; cap them globally (e.g., retries ≤ 10% of requests), or a bad hour becomes a bad day.

Hands-on: build the flaky world (15 min)

New file flaky.py — a payment server that's healthy 60% of the time, plus the (c)-case where work succeeds but the reply dies:

import random, uuid
from fastapi import FastAPI, Header, HTTPException

app = FastAPI()
charges: dict[str, dict] = {}          # idempotency-key → result

@app.post("/charge")
def charge(amount: int, idempotency_key: str = Header()):
    if idempotency_key in charges:               # seen it: replay stored answer
        return charges[idempotency_key]
    if random.random() < 0.4:                    # chaos: fail BEFORE work
        raise HTTPException(status_code=503)
    result = {"charge_id": str(uuid.uuid4()), "amount": amount}
    charges[idempotency_key] = result            # work done + recorded together
    if random.random() < 0.3:                    # chaos: fail AFTER work (case c!)
        raise HTTPException(status_code=503)
    return result

And a client that survives it, client.py:

import httpx, time, random, uuid

def charge_with_retries(amount: int) -> dict:
    key = str(uuid.uuid4())                      # ONE key for the whole operation
    for attempt in range(5):
        try:
            r = httpx.post("http://localhost:8000/charge",
                           params={"amount": amount},
                           headers={"idempotency-key": key}, timeout=2.0)
            if r.status_code == 200:
                return r.json()
        except httpx.TimeoutException:
            pass
        time.sleep((2 ** attempt) + random.random())   # backoff + jitter
    raise RuntimeError("gave up after 5 attempts")

print(charge_with_retries(50))

Run the server (fastapi dev flaky.py), then python client.py a few times (pip install httpx first). Watch the win: even when the server 503s after recording the charge, your retry gets the same charge_id back — one key, one charge, however many attempts. Then comment out the idempotency_key in charges check and run again: duplicate charge_ids. You've reproduced the industry's most expensive bug class and its fix in 40 lines.
