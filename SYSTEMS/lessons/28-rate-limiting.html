Rate Limiting & Load Shedding

One skill, learned today: bound how fast each client may hit you — token bucket in your own code — and shed load gracefully when everyone arrives at once.

Why this matters

"Design a rate limiter" is a top-five interview question in its own right — and every API you'll ever ship (or call: remember the 429s from lesson 02?) has one. It's also half of a deeper production creed: a system that can't say no will be destroyed by yes.

Token bucket: the algorithm to know cold

Each client gets a bucket holding up to burst tokens, refilled at rate tokens/second. A request costs one token: token available → allow; bucket empty → 429 with a Retry-After header. The beauty: it permits natural burstiness (a page load firing 10 quick calls) while enforcing the long-run average [Hello Interview: rate limiter]. Alternatives to name and drop: fixed window (simple; 2× bursts at window edges), sliding window log (exact; costly), sliding window counter (the practical middle). Token bucket is the default answer.

Where does the state live? For one server, a dict. For a fleet (07: any request, any server), the buckets must be shared — Redis, with the check-and-decrement done atomically (a Lua script or atomic ops — the lost-update lesson 14 again, at the cache layer!). Big fleets often accept per-node local limiting (limit ÷ N per node) — slightly loose, zero shared-state latency: a classic accuracy-vs-cost call, eventual-consistency thinking (18) applied to counters.

Hands-on: token bucket in 25 lines (15 min)

import time

class TokenBucket:
    def __init__(self, rate: float, burst: int):
        self.rate, self.burst = rate, burst
        self.tokens, self.last = float(burst), time.monotonic()

    def allow(self) -> bool:
        now = time.monotonic()
        self.tokens = min(self.burst, self.tokens + (now - self.last) * self.rate)
        self.last = now
        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False

bucket = TokenBucket(rate=2, burst=5)        # 2 req/s sustained, bursts of 5
for i in range(12):
    print(f"req {i:2d}: {'ALLOW' if bucket.allow() else '429 '}")
    time.sleep(0.2)                          # 5 req/s attempted — too fast!

Run it: the first 5 fly (burst), then allows come only as the 2/s refill drips in — attempted 5/s throttled to the contract. Lazy refill (compute on demand from elapsed time — no background timer) is the trick worth remembering; it's how real implementations work. Wire it into your notes API as a FastAPI dependency keyed by client IP if you want the full experience.

Load shedding: rate limiting's system-wide sibling

Rate limiting is per-client fairness. Load shedding is self-preservation: when total load nears capacity, reject early and cheaply (fast 503 + Retry-After at the front door) rather than accept work you'll fail slowly — because overload latency climbs, callers time out and retry (22), adding load exactly when you can least afford it: the retry storm death spiral. Shedding breaks the spiral [Google SRE: handling overload]. Refinements that mark seniority: shed by priority (drop analytics pings, keep checkouts) and put degraded modes between "full service" and "shed" (serve cached/stale, disable expensive features — graceful degradation, expanded in 36).

AI-lab tie-in

LLM APIs are rate limiting's most visible modern deployment — but with a twist you now appreciate: limits are enforced in tokens per minute, not requests, because cost scales with tokens. Same bucket algorithm, cost-proportional debiting (a 4,000-token request debits 4,000). And when GPU fleets saturate, priority shedding decides: batch jobs wait, interactive chats go first.
