Lesson 0027 · Phase 2 · ~30 minutes

Backtracking II: Constraints & Pruning

Today's skill: make exponential search tractable by killing doomed branches early — constraints as weapons, not obstacles.

Why this, now

Lesson 0026 generated everything. Real problems want solutions satisfying constraints — and the constraint is a gift: every condition a partial path already violates lets you abandon 2ᵏ descendants unexplored. Pruning is what separates "correct but times out" from "accepted."

The principle

A leaf-check validates complete paths; a prune rejects partial ones. Same answers, wildly different work — rejecting one node at depth d saves the entire subtree below it. The craft: push every check as high up the tree as it can correctly go.

Worked: Combination Sum

Candidates [2,3,6,7], target 7 — all combinations summing to 7 (reuse allowed).

def combination_sum(candidates, target):
    candidates.sort()                      # enables prune #2
    result, path = [], []

    def backtrack(start, remaining):
        if remaining == 0:                 # exact hit: a solution leaf
            result.append(path[:])
            return
        for i in range(start, len(candidates)):
            c = candidates[i]
            if c > remaining:              # PRUNE: too big — and sorted means
                break                      #   all later candidates are too. break, not continue.
            path.append(c)                 # choose
            backtrack(i, remaining - c)    # explore (i, not i+1: reuse allowed)
            path.pop()                     # un-choose

    backtrack(0, target)
    return result

Three deliberate moves beyond the 0026 template:

  • Prune on partial state: c > remaining abandons the branch before recursing.
  • Sort to upgrade the prune: sorted candidates turn continue (skip one) into break (kill all remaining siblings) — Lesson 0016's "buy structure" applied to search trees.
  • The start index kills duplicates by design: only choosing candidates at ≥ my index means [2,3] is explored but [3,2] never exists. Deduplicating by construction beats deduplicating results.

Worked: Word Search (backtracking on a grid)

Does "SEE" trace a path of adjacent cells in a letter grid? The decision tree's choices are directions; the state to restore is the board itself:

def exist(board, word):
    rows, cols = len(board), len(board[0])

    def backtrack(r, c, i):                    # matching word[i] at (r, c)
        if i == len(word):        return True  # matched everything
        if not (0 <= r < rows and 0 <= c < cols):  return False   # prune: off board
        if board[r][c] != word[i]:             return False        # prune: wrong letter
        saved, board[r][c] = board[r][c], "#"  # choose: mark visited (in place!)
        found = (backtrack(r+1, c, i+1) or backtrack(r-1, c, i+1) or
                 backtrack(r, c+1, i+1) or backtrack(r, c-1, i+1))
        board[r][c] = saved                    # un-choose: restore the cell
        return found

    return any(backtrack(r, c, 0)
               for r in range(rows) for c in range(cols))

Note the shape holds perfectly: choose (mark), explore (4 directions), un-choose (restore). The "#" marking prevents reusing a cell within one path — and restoring it lets other paths use the cell. This is your bridge to Phase 3: a grid is a graph, and this is graph DFS one lesson early.

N-Queens, in one paragraph

The famous showcase: place n queens, none attacking. Choices = "which column gets the queen in row r"; prunes = three sets (columns, both diagonal directions) checked in O(1); un-choose = remove from all three. The sets-as-constraints trick — O(1) conflict checks via hash sets (Lesson 0005) instead of rescanning the board — is the takeaway. Attempt it after the main problems; expect a fight, and enjoy it.

Mission link · ML/AI

Pruning is the universal law of search: alpha-beta pruning cuts game trees exactly like your break; MCTS prunes softly by visit counts; constrained decoding masks invalid tokens — pruning an LLM's output tree at generation time. And SAT/SMT solvers — which verify chips and protocols — are industrial backtracking with learned pruning. Your template scales further than it looks.
