Shortest Paths: BFS & Dijkstra

Today's skill: pick the right shortest-path tool — BFS when steps count equally, Dijkstra's heap-driven BFS when edges have weights.

Why this, now

Dijkstra is the crown jewel of Phase 3 — the algorithm where graph + heap + greedy meet. It's a standard interview medium ("Network Delay Time," "Cheapest Flights"), the backbone of routing and planning, and conceptually: it's your Lesson 20 BFS with one word changed — nearest replaces fewest-hops.

Unweighted: BFS already solves it

When every edge costs 1, BFS's levels ARE distances — first arrival is the shortest path (Lesson 28 promised this; here's the payoff). Track distance with the level trick or store it alongside each node. O(V + E), done. If a problem says "minimum number of steps/moves," you already own the answer — most "shortest path" interview questions are secretly this (word ladders, knight moves, maze escapes).

Weighted: why BFS breaks, and the fix

With weights, fewest-edges ≠ cheapest: A→B direct might cost 10 while A→C→B costs 3. BFS processes in discovery order; we need cheapest-first order. Replace the FIFO queue with a min-heap keyed on distance (Lesson 23) — that's the entire conceptual delta:

import heapq
from collections import defaultdict

def dijkstra(n, edges, src):                 # edges: (u, v, weight), directed
    adj = defaultdict(list)
    for u, v, w in edges:
        adj[u].append((v, w))

    dist = {}                                # final shortest distances
    heap = [(0, src)]                        # (best-known cost, node)
    while heap:
        d, node = heapq.heappop(heap)        # cheapest unfinalized candidate
        if node in dist:                     # stale entry — already finalized
            continue
        dist[node] = d                       # FINAL: nothing cheaper can come
        for nb, w in adj[node]:
            if nb not in dist:
                heapq.heappush(heap, (d + w, nb))   # a candidate route to nb
    return dist                              # unreachable nodes: absent

O(E log E) — each edge can push once; each pop costs log. (Stated as O(E log V) in textbooks; either is accepted with the reasoning.)

Why greedy is safe here — the one argument to own

When the heap pops node x at distance d, every other candidate path costs ≥ d (the heap says so), and — here's where weights matter — extending any path only makes it longer, because weights are non-negative. So no future route to x can beat d: finalize it. That's the invariant; it's also exactly why Dijkstra fails with negative edges — extending could then make paths cheaper, and early finalization becomes premature. (Negative weights → Bellman-Ford: relax all edges V−1 times, O(V·E). Know its name and when it's needed; implementing it is rarely asked.)

Two implementation notes that separate working code from almost-working code:

  • Lazy deletion: a node may sit in the heap several times (one per discovered route). The if node in dist: continue skip discards stale entries — simpler than updating heap entries in place (which heapq can't do).
  • Finalization ≠ discovery: dist is written exactly once per node, at pop time — not at push time. Writing at push time is the classic subtle bug.

Mission link · ML/AI

Dijkstra generalizes to A* — add a heuristic estimate of remaining cost to the heap key — which is the classic planning algorithm in robotics and games, and the conceptual ancestor of guided search in agents: beam search (Lesson 23) is best-first search with a learned scorer and a width limit. "Explore the most promising frontier node first, where promising = cost so far + estimated cost to go" is a sentence that describes Dijkstra, A*, and a modern agent's tree search alike.
