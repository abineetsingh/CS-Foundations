State & the Database

One skill, learned today: move your API's state out of process memory into a database, and explain why every production system draws that line.

Why this matters

"Where does state live?" is the recurring question of systems design — half of every interview diagram is an answer to it. The rule you internalize today (processes are disposable, data is not) is what later makes scaling (0007), statelessness, and replication (0015) make sense.

Memory is not storage

Lesson 0003 ended with your notes vanishing on restart. Generalize the failure: process memory dies with the process — on crash, on deploy, on the machine rebooting. And it's invisible to any second process. What your application knows must be split in two:

KindExamplesWhere it may live
Scratch / deriveda parsed request, a loop variable, a cache entryprocess memory — fine to lose
State (the truth)users, notes, orders, balancesa database: a separate program whose whole job is durability

A database earns its keep four ways: it writes to disk and survives restarts (durability), it lets many clients read/write the same data without corruption (concurrency), it finds rows fast (indexes — lesson 0013), and it can enforce rules about the data (constraints). A dict gives you none of these.

SQLite: a real database in the standard library

SQLite is a full SQL database that runs inside your process and stores everything in one file — the most deployed database on Earth (it's in every phone and browser). No server to install: Python ships with it [python docs: sqlite3]. Client-server databases like Postgres arrive in phase 2; the SQL is the same.

Replace the dict in main.py:

import sqlite3
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()

def db():
    conn = sqlite3.connect("notes.db")
    conn.row_factory = sqlite3.Row
    return conn

with db() as conn:                    # runs at startup: create the schema
    conn.execute("""CREATE TABLE IF NOT EXISTS notes (
        id   INTEGER PRIMARY KEY AUTOINCREMENT,
        text TEXT NOT NULL
    )""")

class NoteIn(BaseModel):
    text: str

@app.get("/notes")
def list_notes(limit: int = 20):
    with db() as conn:
        rows = conn.execute("SELECT id, text FROM notes LIMIT ?", (limit,))
        return [dict(r) for r in rows]

@app.post("/notes", status_code=201)
def create_note(note: NoteIn):
    with db() as conn:
        cur = conn.execute("INSERT INTO notes (text) VALUES (?)", (note.text,))
        return {"id": cur.lastrowid, "text": note.text}

@app.get("/notes/{note_id}")
def get_note(note_id: int):
    with db() as conn:
        row = conn.execute("SELECT id, text FROM notes WHERE id = ?", (note_id,)).fetchone()
    if row is None:
        raise HTTPException(status_code=404, detail="note not found")
    return dict(row)

@app.delete("/notes/{note_id}", status_code=204)
def delete_note(note_id: int):
    with db() as conn:
        conn.execute("DELETE FROM notes WHERE id = ?", (note_id,))
The ? placeholders are law

Never build SQL with f-strings. f"… WHERE id = {note_id}" lets a malicious input rewrite your query — SQL injection, still a top real-world vulnerability [OWASP Top 10]. Placeholders (?) make the database treat input as pure data, never as SQL. Full security lesson: 0037.

AI-lab tie-in

Agents have exactly this split: the model's context window is process memory — scratch space, gone when the session ends. Anything an agent must remember (conversation history, learned facts, task queues) gets written to a store. "What goes in context vs what goes in the database" is the agent-infrastructure version of today's lesson.
