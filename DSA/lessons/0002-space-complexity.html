Space Complexity: The Memory Side of Big-O

Today's skill: state how much extra memory a function uses, using the same Big-O vocabulary you learned for time.

Why this, now

Interviewers always ask for both: "time and space complexity?" Answering only time is the most common beginner tell. The good news: you already know the vocabulary — same classes, different resource.

What counts as "space"

Space complexity measures the extra memory your algorithm allocates, as a function of input size n. Two conventions matter:

  • Auxiliary space — memory beyond the input itself. This is what interviewers mean by default.
  • The input doesn't count against you; a new copy of it does.

Reading space from code

You allocate…Space
A few variables (count, best, i)O(1)
A list/dict/set that can grow to n entriesO(n)
A slice or copy of the input (arr[:], sorted(arr))O(n)
An n × n grid or tableO(n²)
Recursive calls d levels deep (call stack)O(d) — more in Lesson 0017

Example 1 — O(1) space

def reverse_in_place(arr):
    left, right = 0, len(arr) - 1
    while left < right:
        arr[left], arr[right] = arr[right], arr[left]
        left += 1
        right -= 1

Time O(n) — we touch every element. Space O(1) — two integer variables, no matter how long the list. "In place" is interview shorthand for O(1) extra space.

Example 2 — O(n) space

def reversed_copy(arr):
    result = []
    for x in arr:
        result.insert(0, x)   # (also slow — why? see quiz)
    return result

Same job, but result grows to n elements: space O(n).

The trade-off — spending space to buy time

This is the deepest idea in the lesson. Recall has_duplicate from Lesson 0001: comparing all pairs took O(n²) time and O(1) space. Watch what happens when we spend memory:

def has_duplicate(arr):
    seen = set()
    for x in arr:
        if x in seen:          # O(1) average — Lesson 0005 explains why
            return True
        seen.add(x)
    return False

Now: O(n) time, O(n) space. We bought a whole complexity class of time with memory. Most interview "optimizations" are exactly this trade, made deliberately. When you propose it, say the trade out loud.

Mission link · ML/AI

The time↔space trade runs all through ML systems. A KV cache spends O(n) memory per sequence to avoid recomputing attention keys/values each token — memory buying time. Gradient checkpointing is the reverse trade: recompute activations during backprop to fit bigger models in GPU memory — time buying memory. Same currency exchange you just learned.
