The Interview Loop: A Repeatable Protocol

Today's skill: a six-step protocol that turns "solve this under pressure" into a rehearsed procedure — because interviews grade the process, not just the answer.

Why this, now

You now know the algorithms. Interviews measure something adjacent: can you communicate problem-solving — thinking aloud, structuring ambiguity, verifying honestly? Candidates fail with correct code silently written, and pass with imperfect code lucidly reasoned. The protocol below is distilled from the Tech Interview Handbook's battle-tested framework.

The six steps (45-minute budget in brackets)

1. Clarify — before ANY solving [3–5 min]

Restate the problem in your own words. Then ask: input size and ranges? duplicates? sorted? empty/null cases? what to return on failure? "Can I assume the array is non-empty, or should I handle that?" — every question signals engineering maturity, and input size hints the target complexity (n ≤ 20 → exponential fine (26); n ≈ 10⁵ → O(n log n) or better; n ≈ 10⁸ → O(n) or O(log n)).

2. Work an example by hand [2–3 min]

A concrete input, worked on the shared doc/whiteboard. This isn't for the interviewer — it's how your pattern recognition (all 40 lessons of it) actually fires.

3. State the brute force — out loud, don't code it [2 min]

"Brute force: check all pairs, O(n²). Let me look for better." This banks a working answer, sets the baseline, and — critically — gives you the gap to attack: what's redundant in the brute force? (Recompute → cache/window. Search → hash/sort/binary. All paths → DP.)

4. Optimize, narrating the pattern hunt [5–10 min]

Run your tells aloud: "Sorted input — two pointers? … 'Fewest steps' — BFS? … Budget + take-or-skip — knapsack?" Wrong turns spoken are fine (they show method); silence is the killer. Agree on the approach with the interviewer before writing code — they'll often steer you off a doomed path, which is help you only get by talking.

5. Code it — clean, top to bottom [10–15 min]

Real variable names (left, not l), helper functions for sub-tasks, and keep narrating at the level of chunks ("now the shrink-loop maintains the invariant…"). If stuck on a detail, write a TODO and move — a complete skeleton with one gap beats a perfect fragment.

6. Verify before declaring victory [5 min]

Trace your code — line by line, updating variables on the example from step 2 as if you didn't write it. Then edge cases: empty, single element, duplicates, extremes. Then, unprompted: "Time is O(n log n) for the sort dominating the O(n) sweep; space is O(n) for the table — where n is the number of intervals." Complexity, both kinds, with n defined (the Lesson 01 phrasing, now habitual). Saying "done" without this step is the single most common self-inflicted rejection.

Mission link · ML/AI

AI-lab interviews leans harder on this protocol than big-tech ones: research culture prizes reasoning transparency — the same quality you'd want in a model's chain of thought. Practically: OpenAI/Anthropic loops mix classic DSA rounds with practical coding (Lesson 44), and BOTH grade the think-aloud protocol. This lesson is the transferable layer.
