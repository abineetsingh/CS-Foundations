Lesson 0029 · ~25 minutes · Systems Design · Phase 3

Realtime Delivery

One skill, learned today: choose correctly among polling, long-polling, SSE, and WebSockets when the server has news for the client — and know what holding a million connections costs.

Why this matters

HTTP's whole design (lesson 0001) is client-asks-server-answers. But chat messages, notifications, live scores, and streaming LLM tokens are news the server learns first. Every "design chat/notifications/live-X" interview turns on this choice within minutes — and it sets up lesson 0042's chat drill.

The four options, weakest to strongest

TechniqueHowLatencyChoose when
Short pollingclient asks every N seconds: "anything new?"up to N sN can be large (badge counts); dumbest thing that works — start here
Long pollingserver holds the request open until news arrives (or ~30s timeout), answers, client immediately re-asks~instantnear-realtime through plain HTTP; the classic pre-WebSocket answer
SSE (server-sent events)one long-lived HTTP response that never ends; server appends events as text [MDN: SSE]instantone-directional server→client streams: notifications, feeds, LLM tokens
WebSocketHTTP handshake upgrades the connection to a persistent, bidirectional message channel [MDN: WebSockets]instant, both waysgenuine two-way traffic: chat, multiplayer games, collaborative editors

The selection habit interviewers reward: climb only as high as the requirement forces. Notifications don't need WebSockets (SSE suffices — one direction). Chat does (typing, sending, receiving). Saying "polling is fine here" when it is — that's engineering; reflexively saying "WebSockets" is fashion.

What a million open connections does to your architecture

Persistent connections quietly break two phase-1 assumptions:

  • Statelessness bends (0007): an open socket IS state — user 42's messages must reach the specific server holding her socket. The standard shape: a dedicated, connection-holding gateway tier (idle connections are cheap — mostly memory, ~10s of KB each; async servers hold 100k+ per node), with a registry (Redis: user → gateway node) or pub/sub (0024!) routing messages to the right box. App logic behind it stays stateless.
  • LB behavior changes (0008): connections are long-lived, so "least connections" balances poorly at connect-time bursts and reconnect storms after a gateway dies become a thundering herd (0010) — clients must reconnect with jittered backoff (0022, everywhere forever).
  • Delivery is still at-least-once (0025): sockets drop; clients miss messages while reconnecting. Real systems pair the push channel with a pull catch-up ("give me everything since sequence 8412") — the event-log-with-offsets idea (0024), per conversation.
AI-lab tie-in

Watch a Claude response stream in: that's SSE — one long HTTP response, tokens appended as generated (the API literally sends event: content_block_delta lines). One-directional fits perfectly: you asked once; the server has a stream of news. The whole voice-agent world (bidirectional audio) upgrades to WebSockets for exactly the reason in the table. You now know why each choice was made.
