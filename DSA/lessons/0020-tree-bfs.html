Binary Trees: Breadth-First Search

Today's skill: traverse a tree level by level with a queue — and the level-size trick that makes "per-level" questions easy.

Why this, now

BFS completes your traversal toolkit: DFS goes deep (stack), BFS goes wide (queue) — Lesson 0011's promise that the container is the algorithm, kept. Any question with "level," "nearest," or "minimum depth" in it is BFS bait.

The skeleton

from collections import deque

def bfs(root):
    if not root: return
    q = deque([root])
    while q:
        node = q.popleft()          # oldest discovery first (FIFO)
        visit(node)
        if node.left:  q.append(node.left)
        if node.right: q.append(node.right)

Visits: 1, 2, 3, 4, 5 on Lesson 0019's tree — top to bottom, left to right. O(n) time; space is the queue's peak, which is the widest level — up to O(n) (the bottom level of a full tree holds n/2 nodes). Note the inversion: DFS space tracks height, BFS space tracks width. Deep narrow tree → BFS wins; shallow bushy tree → DFS wins.

The level-size trick

The skeleton blurs level boundaries. To process levels as units — "level order traversal," "right side view," "average per level" — snapshot the queue size before draining exactly that many:

def level_order(root):
    if not root: return []
    q, levels = deque([root]), []
    while q:
        size = len(q)                    # everyone in q RIGHT NOW is one level
        level = []
        for _ in range(size):            # drain exactly this level
            node = q.popleft()
            level.append(node.val)
            if node.left:  q.append(node.left)   # children queue for NEXT level
            if node.right: q.append(node.right)
        levels.append(level)
    return levels

Why it works: when the loop starts an iteration, the queue contains precisely one full level (invariant — provable by induction, Lesson 0017 style). Children appended during the drain sit behind the snapshot boundary. This one trick converts a dozen LeetCode problems into "level_order plus one line": last element of each level = right side view; sum(level)/len(level) = averages; first level with a leaf = minimum depth.

Choosing DFS vs BFS

Question smells like…UseWhy
levels, width, zigzag, side viewsBFSlevels are its native unit
nearest/shallowest anythingBFSfirst hit is provably closest — no need to explore deeper
subtree computations, paths, "combine children"DFSpost-order's ask-and-combine (Lesson 0019)
either worksDFSfive lines of recursion beats queue bookkeeping
Mission link · ML/AI

"Explore cheapest/nearest candidates first, level by level" is the ancestor of serious search: beam search (Lesson 0023) is BFS that keeps only the k most promising nodes per level — the pruning that makes LLM decoding tractable. Game-tree agents (MCTS in AlphaGo) blend BFS-like frontier expansion with learned value estimates deciding which levels deserve width.
