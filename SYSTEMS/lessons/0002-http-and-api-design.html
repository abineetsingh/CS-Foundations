HTTP & API Design

One skill, learned today: given a feature, name the endpoints — the right method, the right path, the right status codes.

Why this matters

"Design the API" is an explicit early step in nearly every system design interview — usually the first thing you write after requirements. And in real work, the API is the contract everything else hangs off. This vocabulary must be automatic before we build anything.

Methods: verbs with promises

Lesson 0001 showed a request = method + path. The method isn't decoration — each one carries a promise the client and server rely on [MDN: HTTP methods]:

MethodMeaningPromise
GETread a resourcesafe: changes nothing, can be cached or retried freely
POSTcreate / do somethingno promise — repeating it may create two things
PUTreplace a resource entirelyidempotent: doing it twice = doing it once
PATCHupdate part of a resourcepartial update; idempotency depends on the patch
DELETEremove a resourceidempotent: deleting twice still leaves it deleted

Idempotent is a word you'll use in every later phase — retries, queues, and payments all hinge on it. File it carefully: safe to repeat, same end state.

Paths: nouns, not verbs

The REST convention: paths name resources (nouns, usually plural); the method supplies the verb. Predictability is the whole point — a caller should be able to guess your API:

GET    /notes            list notes        (query params filter: /notes?author=42&limit=20)
POST   /notes            create a note     (body carries the JSON)
GET    /notes/17         read note 17
PATCH  /notes/17         edit note 17
DELETE /notes/17         delete note 17
GET    /users/42/notes   notes belonging to user 42 (nesting, one level max)

Anti-patterns to never emit: verbs in paths (/createNote, /getNotes), state changes via GET (/notes/17/delete), and unbounded lists — always paginate with ?limit= and a cursor or offset, because "return all rows" is a future outage [Hello Interview].

Status codes: the three families that matter

You met 200/404/500 in lesson 0001. The full working set is small [MDN: status codes]:

CodeUse it when
200 OK / 201 Created / 204 No Contentsuccess: read / created (with a Location header) / succeeded, nothing to return (e.g. DELETE)
400 / 401 / 403 / 404 / 409 / 429client's fault: malformed / not logged in / logged in but not allowed / doesn't exist / conflict (duplicate) / too many requests (rate limited — lesson 0028)
500 / 503server's fault: unhandled failure / temporarily overloaded or down for deploy

The 401-vs-403 pair is a favorite quick interview probe: 401 = "who are you?" (authentication), 403 = "I know who you are, and no" (authorization). More in lesson 0037.
