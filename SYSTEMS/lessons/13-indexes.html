Indexes

One skill, learned today: predict whether a query scans or seeks, read the plan that proves it, and pay the true price of each index you add.

Why this matters

"Add an index on that column" is the single most common database fix in industry, and "what would you index?" reliably follows "design the schema" in interviews. It's also your DSA knowledge cashing in directly: an index is binary search (DSA 14) living on disk.

Scan vs seek

Without help, SELECT * FROM tweets WHERE user_id = 2 must check every row — a full table scan, O(n) over possibly billions. An index is a separate, sorted structure mapping column value → row location. Sorted means searchable in O(log n) — millions of rows in a few hops. Databases use a B-tree: a binary search tree widened so each node holds hundreds of keys, keeping the tree only 3–4 levels deep — because each level is a disk read, and lesson 05 taught you what those cost [DDIA ch. 4].

Hands-on: catch the planner in the act

In your twitter.db from lesson 12, ask SQLite to explain itself:

sqlite3 twitter.db
EXPLAIN QUERY PLAN SELECT * FROM tweets WHERE user_id = 2;
--> SCAN tweets                                  ← the bad word

CREATE INDEX idx_tweets_user ON tweets(user_id);

EXPLAIN QUERY PLAN SELECT * FROM tweets WHERE user_id = 2;
--> SEARCH tweets USING INDEX idx_tweets_user…   ← the good word

SCAN → reads everything; SEARCH … USING INDEX → walks the B-tree. Every serious database has this facility (EXPLAIN ANALYZE in Postgres); checking the plan — not guessing — is the professional habit [SQLite: EXPLAIN QUERY PLAN] [Use The Index, Luke].

What indexes cost, and composite keys

  • Writes: every INSERT/UPDATE must also update every index on the table. Five indexes ≈ six writes per insert. Index what you query, not everything — a write-heavy table wants few indexes.
  • Space: each index is a full sorted copy of its columns.
  • Composite indexes obey the leftmost-prefix rule: INDEX (user_id, created_at) serves WHERE user_id = ? and WHERE user_id = ? ORDER BY created_at — but is useless for WHERE created_at > ? alone. It's a phone book: sorted by (last, first) — you can't efficiently find everyone named "Ada" by first name. Column order is a design decision.

For the timeline query, the winning move is CREATE INDEX ON tweets(user_id, created_at) — it finds a followee's tweets already in time order, no sort step. Matching the index to the exact (filter, order) shape of the hot query is the craft.

AI-lab tie-in

Vector databases confront the identical problem one level up: exact nearest-neighbor search over embeddings is a full scan, so ANN indexes (HNSW — a navigable graph, cousin of the B-tree idea) trade a little recall for O(log-ish) search.
