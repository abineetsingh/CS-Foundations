Glossary

Every term the course defines, in one place, phase by phase. Skim before interviews; terms are never redefined differently in later lessons.

Phase 1 — Foundations

TermMeaning
ClientThe side that initiates a request: browser, app, or a Python script calling an API. A role, not a device.
ServerA program that listens for requests and answers them. Not the hardware — one machine can run many servers.
BackendEverything server-side powering an application: API servers, databases, caches, queues, and the infrastructure beneath.
RequestClient→server message: method (GET, POST…), path, headers, optional body.
ResponseThe server's answer: status code, headers, usually a JSON body.
HTTPThe request/response protocol clients and servers speak on the web.
IP addressNumeric machine address (203.0.113.7) — what packets are delivered to.
PortNumber (0–65535) selecting which program on a machine receives traffic. HTTPS: 443; dev servers: 8000 etc.
DNSThe internet's phone book: name → IP address. Always the first stop.
localhost"This machine" (127.0.0.1). No DNS, no network — how you test locally.
Status code2xx success (200/201/204), 4xx client's fault (400/401/403/404/409/429), 5xx server's fault (500/503).
EndpointOne path+method a server answers (GET /users/42). An API is a set of endpoints.
IdempotentSafe to repeat: doing it twice leaves the same end state as once (PUT, DELETE — not POST). The load-bearing word of phase 3.
StateData that must outlive a request or process — users, orders, balances. Truth lives in databases; process memory is scratch.
DatabaseThe program whose job is durable, concurrent, indexed, constrained storage of truth.
LatencyTime for one operation, start to finish. Memory: ns; SSD: µs; network: ms — see the latency card.
QPSQueries per second. Envelope math: daily actions ÷ 10⁵; peak ≈ 2–10× average.
Vertical scalingBuy a bigger machine. Zero code changes; hard ceiling; still a single point of failure.
Horizontal scalingAdd more machines. No ceiling, survives failures — requires stateless services.
StatelessA service keeping no client-visible state in its own memory between requests; truth externalized. Any server can serve any request.
Session"Who is logged in" state. Externalize to a shared store or carry as a signed token — never keep in one server's memory.
Load balancerFronts a fleet with one address; spreads requests across healthy backends (round robin / least connections); health-checks them. L4 sees IPs/ports; L7 sees HTTP.
Reverse proxyA server standing in front of others, relaying traffic (nginx, HAProxy). LBs, gateways, and CDN edges are all reverse proxies.
CacheSmall fast store of copies of expensive-to-fetch data. Every cache trades freshness for speed.
TTLTime-to-live: a cache entry's self-destruct timer. Bounds memory and staleness at once.
Hit ratioFraction of reads served from cache (h). Effective latency = h·t_hit + (1−h)·t_miss; the source sees only (1−h) of traffic.
Thundering herd / stampedeA hot cache key expires; thousands miss simultaneously and charge the database. Fixes: per-key coalescing, TTL jitter, stale-while-revalidate.
CDNPlanet-scale cache fleet (Cloudflare, CloudFront). Users hit a nearby edge instead of your origin. Controlled via Cache-Control.
EdgeA CDN's point of presence near users (~10 ms away), vs your "origin" servers.

Phase 2 — The data layer

TermMeaning
Primary keyA row's unique, permanent identifier. Also enforces uniqueness.
Foreign keyA column holding another table's primary key — an enforceable typed pointer. Joins dereference it.
JoinMatching rows across tables by key. Many-to-many needs a join table of two FKs.
NormalizationEvery fact stored once. Denormalization = deliberate duplication for read speed, paid at update time. Normalize by default; denormalize by evidence.
IndexA sorted structure (B-tree) mapping column value → row location: O(log n) lookups, taxing every write. Composite indexes obey the leftmost-prefix rule.
B-treeA search tree widened so each node holds hundreds of keys — 3–4 levels for millions of rows; each level is one disk read.
Full table scanReading every row because no index fits — O(n). EXPLAIN says SCAN; the fix says SEARCH … USING INDEX.
TransactionBEGIN…COMMIT: multi-step change made all-or-nothing (ACID: atomic, consistent, isolated, durable). Watch for lost updates under weak isolation.
ReplicationLive copies of data on multiple machines. Leader takes writes, streams changes to followers; reads scale across copies.
Replication lagFollowers run slightly behind the leader (async replication). Causes stale reads and missing-own-write bugs; fixed by read-your-writes routing.
FailoverLeader dies → promote a follower. Safe only when a quorum ratifies it (else: split brain). Async lag risks losing last acked writes.
Sharding / partitioningSplitting rows across machines by a shard key. Golden rule: the hot query touches one shard. Range vs hash; beware hot keys (celebrities).
Consistent hashingKeys and nodes on a hash ring; a key belongs to the next node clockwise. Adding a node moves ~1/N of keys (vs ~all under % N). Virtual nodes smooth arcs.
Object storagePlanet-scale KV store for files (S3, GCS): flat key → bytes, 11-nines durable, HTTP-native. Metadata in the DB; bytes here; CDN in front.
BlobBinary large object — images, video, model checkpoints. Never store in the database.
Presigned URLA short-lived signed grant ("PUT exactly this key for 10 min") letting clients upload/download directly to object storage, bypassing your servers.
(Network) partitionThe network splits; both sides alive but mutually unreachable. CAP: during one, each side chooses consistency (refuse) or availability (serve possibly-stale).

Phase 3 — Distributed systems

TermMeaning
Idempotency keyClient-minted unique id per operation (not per attempt); server stores processed ids and replays results for repeats. Makes retries safe.
Retry stormOverload → timeouts → retries → more overload. Broken by backoff+jitter, retry budgets, and load shedding.
Message queueDurable buffer between producers and workers (SQS, RabbitMQ). Each job to one worker; ack-after-work; visibility timeout redelivers on crash.
WorkerA consumer process draining a queue at its own pace. Must be idempotent (redelivery happens). Scale on queue depth.
BackpressureSignaling "slow down" upstream when consumers can't keep up. Queues absorb bursts, not permanent rate mismatch — bound them.
Event log / streamAppend-only ordered sequence of facts (Kafka). Consumers read without consuming, each tracking its own offset; retained and replayable; partitioned by key (order holds within a partition).
OffsetA consumer's bookmark in an event log — advance, or rewind to replay history (idempotently!).
Event-driven architectureServices react to published facts rather than calling each other — one write, N independent consumers; loose coupling.
QuorumA majority (⌊N/2⌋+1). Two majorities of one set must overlap — hence at most one elected leader, ever. Odd cluster sizes; minority side stalls.
Split brainTwo leaders accepting writes after a careless failover during a partition. Prevented by quorum elections (Raft).
ConsensusProtocols (Raft, Paxos) for a cluster to agree despite failures. Used bottled: etcd/ZooKeeper for leader election, locks, config — small critical facts only (it's expensive).
SagaMulti-service operation as a sequence of local transactions, each with a compensating undo (refund, release), run in reverse on failure. Eventually consistent mid-flight ("pending").
Outbox patternWrite the event as a row in the same local transaction as the business change; a relay publishes it at-least-once. Closes the commit-then-crash-before-publish gap.
Load sheddingNear capacity, reject excess early and cheaply (fast 503/429) — by priority — rather than fail everything slowly. Breaks retry storms.
MonolithOne deployable unit (× N stateless copies). Function calls, shared transactions, one deploy train. The right default for young systems.
MicroservicesIndependently deployed services, each owning its data, talking over the network. Buys team autonomy; costs 0022–0027 on every internal edge. Shared database = distributed monolith.
API gatewayThe fleet's one front door: auth, rate limiting, routing. Clients never learn your org chart.

Phase 4 — Production & infrastructure

TermMeaning
ContainerA normal process isolated by namespaces (private view) and cgroups (resource caps), carrying its own userspace. Not a VM; starts in ms; filesystem is ephemeral.
ImageThe frozen, layered, immutable artifact a container runs from (myapi:v42), stored in a registry. Immutability makes rollback = run the old tag.
OrchestrationKubernetes-class systems doing scheduling, self-healing, scaling, service discovery — via declare-desired-state + reconcile loops. Cluster truth in etcd.
CI/CDEvery commit → automated test/build pipeline → verified immutable artifact → staged rollout. No artifact ships unverified.
Percentile (p50/p99)Latency is a distribution: p50 = median, p99 = worst 1%. Averages lie; tails compound under fan-out (1−0.99¹⁰ ≈ 10%). Alert on p99.
SLO / error budgetA committed target on a measured SLI ("99.9% of reads < 200 ms monthly"). 100%−SLO = the error budget: spend it shipping, freeze when it's gone. Alert on burn rate.
Circuit breakerWraps a dependency; too many failures → opens (instant local failure, no calls) → half-open probes → closes. Pairs with bulkheads (per-dependency pools) and designed fallbacks (graceful degradation).
RegionA geographic cluster of datacenters. Multi-region = latency/DR project with hard consistency tradeoffs (~50–150 ms apart).
Availability zone (AZ)One independent datacenter within a region (~1 ms apart). Spreading replicas across AZs is the default cheap fault-tolerance.
