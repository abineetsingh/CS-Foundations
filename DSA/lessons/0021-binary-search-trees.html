Binary Search Trees

Today's skill: exploit the BST invariant — search in O(h), and reason about problems with value bounds instead of exhaustive checks.

Why this, now

A BST is binary search (Lesson 0014) frozen into a data structure: the sorted array's "discard half per comparison," but with O(h) insertion too. It's also the sharpest test yet of invariant thinking — the classic "validate a BST" question is famous precisely because the obvious solution is wrong.

The invariant

At every node: all values in the left subtree < node < all values in the right subtree. Not just the children — the entire subtrees.

        8
       / \
      3   10          search(6): 6 < 8 → go left. 6 > 3 → go right.
     / \    \         6 > 4? no—wait: subtree of 3's right is {4,6,7}…
    1   6    14       each comparison discards a whole subtree.
       / \
      4   7
def search(node, target):
    if not node:               return None
    if target == node.val:     return node
    if target < node.val:      return search(node.left, target)
    else:                      return search(node.right, target)

O(h) time — one comparison per level, one subtree discarded per comparison. Balanced: O(log n). Degenerate (inserted in sorted order — a chain): O(n). This is the BST's asterisk: the invariant guarantees order, not balance. Self-balancing variants (AVL, red-black) fix this; for interviews you name them and move on — implementing one is not expected.

Two consequences worth owning

1. In-order traversal yields sorted order

Left < node < right, applied recursively, means Lesson 0019's in-order walk visits values ascending: 1, 3, 4, 6, 7, 8, 10, 14. Instantly solves "kth smallest in a BST" (in-order, count to k) and "validate" (in-order, check ascending).

2. Validation needs bounds, not child-checks

#   5          WRONG: check node.left.val < node.val < node.right.val
#  / \         at each node. This tree passes — but 6 > 5 sits in the
# 3   8        left subtree of 5. Local checks miss ancestral limits.
#  \
#   6   ← legal vs parent 3, ILLEGAL vs grandparent 5

def is_valid(node, lo=float("-inf"), hi=float("inf")):
    if not node:
        return True
    if not (lo < node.val < hi):        # the window inherited from ancestors
        return False
    return (is_valid(node.left,  lo, node.val) and    # left: cap above by me
            is_valid(node.right, node.val, hi))       # right: floor below by me

Each node lives in a window (lo, hi) narrowed by every ancestor. Passing constraints down as parameters — rather than answers up — is the complementary recursion pattern to Lesson 0019's ask-and-combine. You now have both directions.

When a BST is the answer

NeedHash mapSorted arrayBST (balanced)
lookupO(1)O(log n)O(log n)
insert/deleteO(1)O(n)O(log n)
range / min / max / successor✗ unorderedO(log n)O(log n)

The niche: ordered operations under change. Static data → sorted array + bisect. No order needed → hash map. Both at once → BST. (Python has no built-in; interviews accept naming sortedcontainers.SortedList.)

Mission link · ML/AI

Databases index with B-trees — BSTs widened for disk pages — which is why your feature-store range queries are O(log n). And decision trees/GBDTs (XGBoost) route inputs root-to-leaf through threshold comparisons exactly like search(); "tree depth" as a regularization knob is literally the h in O(h).
