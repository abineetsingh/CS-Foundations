Partitioning & Sharding

One skill, learned today: split a too-big dataset across machines — choosing the key, the scheme, and surviving the day you add a shard.

Why this matters

Replication copies the whole dataset; it can't help when the dataset itself outgrows one machine — 100 TB doesn't fit anywhere. Sharding is the interview's favorite deep-dive ("how would you shard this?"), and consistent hashing is arguably the single most-asked concept in the genre.

The idea, and the all-important key

Sharding (= partitioning) splits rows across N databases, each holding a slice. Everything downstream follows from one decision: the shard key — which column decides where a row lives. The golden rule: pick the key so your hottest query touches ONE shard. For Twitter-lite, shard tweets by user_id: "all of Grace's tweets" lives together on one machine; timeline assembly reads a handful of shards instead of all of them. Queries that can't use the key must ask every shard (scatter-gather) — slow, and the thing you design to avoid [DDIA ch. 7].

Two famous failure modes

1. The celebrity (hot key): hashing spreads keys evenly, not load. All of Beyoncé's data is still one key on one shard, which melts while its neighbors idle. Mitigations to name: cache the hot entity hard (09), or split the key itself (beyonce#1…#16) and fan back in at read time.

2. Resharding with % N: grow 4 shards to 5 and hash % N changes for 80% of all keys — a planet-sized data migration for one added machine. The fix is the interview classic:

Consistent hashing: place shards and keys on a ring (hash both to positions 0–360°); each key belongs to the next shard clockwise. Adding a shard steals only the arc behind it — ~1/N of keys move instead of ~all. Virtual nodes (each server appears at many ring positions) smooth the arcs. This is how Dynamo-style stores and distributed caches grow without stopping the world [Hello Interview: consistent hashing].

Last piece: someone must know the map. A routing tier (or smart client library) holds "key → shard," and each shard is itself a replicated leader–follower set (15). Sharding × replication compose — that grid is what "the database" means at scale.

AI-lab tie-in

Serving a model too big for one GPU is sharding with physics: tensor parallelism splits layers across GPUs the way hash sharding splits rows. And KV-cache-aware LLM routers use consistent hashing on conversation IDs so a session reliably lands on the replica already holding its cache — the ring, re-invented for GPUs.
