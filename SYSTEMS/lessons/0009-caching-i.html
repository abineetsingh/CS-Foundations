Caching I: The Hierarchy & Cache-Aside

One skill, learned today: add a cache to a read path correctly — and compute exactly how much it buys you.

Why this matters

Your Twitter-lite envelope (lesson 0006) screamed "50,000 read QPS." The single highest-leverage answer in all of systems design is: don't do the same work twice. Every interview design ends up with a cache in it; today you learn to place one deliberately instead of decoratively.

What a cache is, and where they hide

A cache is a small, fast store holding copies of expensive-to-fetch data. It works because of the latency table from lesson 0005 (memory beats network by ~1000×) plus a fact about real traffic: it's brutally skewed — a small set of hot items gets most of the reads (the 80/20 pattern). A request in production may cross five caches, closest-to-farthest:

LayerWhereSaves
Browser cachethe user's devicethe entire network
CDN (lesson 0010)edge servers near the userthe long WAN hop
In-process cachea dict in your API servereverything — but per-server and RAM-bounded
Distributed cacheRedis/Memcached, shared by the fleetthe database query — today's focus
DB buffer poolinside the databasedisk reads (free, automatic)

The shared tier matters most for design: a Redis-style cache is a networked hash map (your DSA lesson 0005 structure, industrialized) doing ~100k+ lookups/sec at sub-millisecond latency, visible to every stateless server in the fleet.

Cache-aside: the pattern to know cold

The dominant read pattern [AWS Builders' Library: caching]. The application (not the cache) owns the logic:

def get_note(note_id):
    key = f"note:{note_id}"
    hit = cache.get(key)
    if hit is not None:               # HIT: ~1 ms, done
        return hit
    row = db.query(note_id)           # MISS: pay the full price
    cache.set(key, row, ttl=300)      # fill, with an expiry (TTL)
    return row

Every entry gets a TTL (time-to-live) — a self-destruct timer. It bounds two things at once: memory growth, and how stale a copy can get. Stale? Yes — if the note is edited during those 300 seconds, readers may see the old version. Every cache is a deliberate trade of freshness for speed. Say that sentence in interviews; handling it properly (invalidation) is lesson 0010.

The arithmetic: hit ratio is everything

Let hit ratio h = fraction of reads served from cache. Effective latency = h × t_hit + (1−h) × t_miss, and — often more important — the database only sees (1−h) of the traffic:

50,000 read QPS, t_hit = 1 ms, t_miss = 10 ms:

h = 90%:  latency ≈ 0.9×1 + 0.1×10 = 1.9 ms   DB sees  5,000 QPS  (ouch → fine)
h = 99%:  latency ≈ 1.09 ms                    DB sees    500 QPS  (a single box shrugs)

Going from 90% → 99% cuts database load another 10× — hit-ratio improvements compound like that, which is why teams obsess over them. What earns high ratios: hot skewed reads, tolerable staleness, keys that repeat. What doesn't: uniformly random reads, write-heavy data, personalized-per-user everything.

AI-lab tie-in

Prompt caching at LLM providers is cache-aside economics verbatim: a repeated prompt prefix is a "hot key," the KV-cache entry is the cached value, the "miss cost" is recomputing attention over thousands of tokens — which is why cached input tokens are billed ~10× cheaper. Same formula, GPU-sized prices.
