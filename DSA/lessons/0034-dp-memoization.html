Dynamic Programming I: Memoization

Today's skill: spot overlapping subproblems, and collapse an exponential recursion to linear with a cache — DP demystified as "recursion + memory."

Why this, now

DP has the scariest reputation of any interview topic, and the reputation is a lie. You already know recursion's three checks (Lesson 0017); memoization adds a dictionary (Lesson 0005). That's the entire entry fee. DP problems dominate the hard end of interview loops — and your two-phase foundation was built precisely so this lesson would feel small.

The disease: overlapping subproblems

def fib(n):                        # correct by the three checks — and doomed
    if n <= 1: return n
    return fib(n - 1) + fib(n - 2)

fib(5)
├─ fib(4)
│   ├─ fib(3)  ←──────┐
│   └─ fib(2)  ←──┐   │  computed AGAIN
└─ fib(3)  ────────│───┘
    ├─ fib(2) ─────┘      …and again
    └─ fib(1)

The recursion tree has ~2ⁿ nodes but only n distinct inputs — the same subproblems recomputed exponentially many times. fib(40) ≈ a billion calls for 40 facts. Contrast merge sort (Lesson 0018): its subproblems are all different slices, so there's nothing to reuse. Overlap is the diagnostic — no overlap, no DP.

The cure: cache by input

def fib(n, memo={}):                   # or: pass memo explicitly
    if n in memo:  return memo[n]      # ← seen this exact subproblem? done.
    if n <= 1:     return n
    memo[n] = fib(n - 1, memo) + fib(n - 2, memo)
    return memo[n]

Each distinct input computed once, O(1) dict ops around it: O(n) time, O(n) space (cache + recursion depth). Exponential → linear by adding three lines. Python even ships the pattern (mention it, then hand-roll in interviews):

from functools import lru_cache

@lru_cache(maxsize=None)
def fib(n):
    if n <= 1: return n
    return fib(n - 1) + fib(n - 2)

The method — four questions that solve DP problems

Climbing Stairs: n steps, climb 1 or 2 at a time — how many distinct ways up?

  1. State — what identifies a subproblem? The step you're standing on. ways(i) = ways to reach the top from step i.
  2. Transition — how do states relate? Your last move was +1 or +2: ways(i) = ways(i+1) + ways(i+2).
  3. Base cases — ways(n) = 1 (arrived), ways(>n) = 0 (overshot).
  4. Cache — memoize on i. Done: O(n).

These four questions are the entire craft — every DP lesson from here refines them. Notice the deep connection: this is backtracking's decision tree (Lesson 0026) where paths reconverge — count the ways through a DAG of states instead of enumerating each path separately. Backtracking lists paths (can't be cached — each path is distinct output); DP counts/optimizes over them (cacheable — only the best/total per state matters). That distinction tells you which tool a problem wants.

Mission link · ML/AI

The KV cache is memoization, verbatim: attention keys/values for a prefix are a pure function of that prefix, so cache them instead of recomputing per token (Lesson 0005's foreshadowing, now formal). Same idea scaled down in lru_cache on a tokenizer, scaled up in prompt caching across requests. "Pure function + repeated inputs → cache" is one of computing's few universal moves, and you now own its theory.
