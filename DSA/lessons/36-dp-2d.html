Dynamic Programming III: Two Dimensions

Today's skill: design states with two coordinates — grid paths and the great sequence-comparison DPs.

Why this, now

When one number can't describe "where you are" in a problem, two can: a grid position, or positions in two different strings. The second family — edit distance, longest common subsequence — is textbook interview material AND the algorithmic core of diff tools, spellcheckers, and DNA alignment. This is DP's power unlocked.

Family 1: grid paths

Unique Paths: from the top-left of an m×n grid, moving only right or down, how many routes to the bottom-right? Four questions: state = cell (r, c); transition = you arrived from above or from the left; bases = the top row and left column (one route each):

def unique_paths(m, n):
    dp = [[1] * n for _ in range(m)]         # first row & column: 1 way
    for r in range(1, m):
        for c in range(1, n):
            dp[r][c] = dp[r-1][c] + dp[r][c-1]   # from above + from the left
    return dp[m-1][n-1]

O(m·n) time — states × O(1) transition, the pricing formula from Lesson 34. Since each row reads only the previous row, the rolling squeeze (Lesson 35) drops space to O(n): keep one row. Variants swap the transition: obstacles zero a cell out; "Minimum Path Sum" turns + into min(...) + grid[r][c]. Same skeleton, dozen problems.

Family 2: two sequences — the crown jewels

Longest Common Subsequence: longest sequence appearing (not necessarily contiguously) inside both strings. The state insight that unlocks every string-pair problem: dp[i][j] = the answer for the first i chars of s and first j chars of t. The transition asks about the last characters:

def lcs(s, t):
    m, n = len(s), len(t)
    dp = [[0] * (n + 1) for _ in range(m + 1)]    # +1: empty-prefix armor (09!)
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if s[i-1] == t[j-1]:                  # last chars match:
                dp[i][j] = dp[i-1][j-1] + 1       #   both consumed, +1
            else:                                 # they can't both be in the LCS:
                dp[i][j] = max(dp[i-1][j],        #   drop s's last char
                               dp[i][j-1])        #   or drop t's
    return dp[m][n]
      ""  a  c  e            s = "abcde", t = "ace"
  ""   0  0  0  0
  a    0  1  1  1            each cell: a subproblem "LCS of prefixes"
  b    0  1  1  1            answer accumulates toward the corner ↘
  c    0  1  2  2
  d    0  1  2  2
  e    0  1  2  3   → LCS = 3 ("ace")

O(m·n) time and space (squeezable to two rows). Edit distance (Levenshtein) is the same table with a different menu: match is free; otherwise 1 + min(insert, delete, replace) = 1 + min(dp[i][j-1], dp[i-1][j], dp[i-1][j-1]). Derive it from LCS rather than memorizing — the interviewer's follow-up is usually exactly that derivation.

Mission link · ML/AI

This table is everywhere in ML history: sequence alignment in bioinformatics (Needleman-Wunsch IS this algorithm), dynamic time warping for audio, the Viterbi algorithm decoding HMMs — the pre-transformer speech stack was 2-D DP. Modern echo: WER (word error rate) for ASR is edit distance; and CTC loss for speech models is computed with exactly this style of prefix-alignment DP. When attention papers say they replaced "alignment," this table is what they replaced.
