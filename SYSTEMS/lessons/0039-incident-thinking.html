Incident Thinking

One skill, learned today: run the incident lifecycle — detect, mitigate, then learn blamelessly — and read real postmortems the way engineers read game film.

Why this matters

Everything in phases 1–4 eventually fails anyway — and how organizations respond separates the great ones from the lucky ones. Interviewers at infrastructure-serious companies ask "tell me about debugging a production issue" and "walk me through this outage"; postmortem literacy is also the single best free education in systems design that exists. This closes phase 4.

The lifecycle: mitigate first, understand later

The incident begins when an SLO-burn alert pages (0035) — not when a user tweets, if you've built well. From there, a discipline that feels backwards until you've lived it [Google SRE: managing incidents]:

StageThe moveThe trap it avoids
Detectalert on symptoms users feel (SLO burn), severity-classified (SEV1: product down → SEV3: degraded)alerting on every internal blip → pager fatigue → real pages ignored
Mitigatestop the bleeding without understanding it: roll back the last deploy (0034 — it's the cause ~most of the time), flip the feature flag, shed load (0028), fail over (0015)debugging root cause while users burn — diagnosis is for daylight; mitigation is for now
Coordinateone incident commander directing; others investigate/communicate; status updates on a cadenceten engineers in a channel all restarting different things (making the timeline unreadable and sometimes the outage worse)
Learnthe blameless postmortem, below — where the real value is extracted"human error, be more careful" → nothing changes → same outage next quarter

The blameless postmortem

A postmortem documents: timeline, impact (users × minutes, in SLO-budget terms — 0035), contributing causes (plural — never one "root cause"; real failures are chains, as 0036 showed), what went well/poorly, and action items with owners. Blameless is the load-bearing word: if the analysis lands on "engineer X made a mistake," it isn't finished — why did the system let one mistake become an outage? Missing canary? No rate limit? Alert that didn't fire? Humans err at a constant rate; systems determine the blast radius. Blame optimizes for hiding information; blamelessness optimizes for surfacing it — and the postmortem's action items are how reliability actually compounds [Google SRE: postmortem culture].

Hands-on: read two real postmortems (15 min)

Your ongoing-practice habit starts now. Read these two classics — with your full toolkit, they read like annotated case law:

  • GitHub, Oct 2018: 43 seconds of network partition (0018!) → split-brain-ish divergence between datacenters (0026!) → 24 hours of careful reconciliation. Watch for: why they chose consistency over availability during recovery.
  • Cloudflare, Jul 2019: one regex → CPU exhaustion → global outage in minutes. Watch for: the deploy-pipeline gap (0034) and how a "safe" config change wasn't.

For each, extract on paper: the trigger, the propagation chain (which 0036 defense was missing?), the mitigation, and two action items you'd have filed. That extraction ritual — trigger → chain → missing defense → fix — is the weekly practice from here on.

AI-lab tie-in

AI products add novel incident classes — a bad model rollout (quality regression rather than errors: harder to alert on; eval-based canaries are the answer, 0034), prompt-injection exploitation in the wild (0037), a runaway agent burning budget in a loop (rate limits + kill switches, 0028's vocabulary). Same lifecycle, same blameless discipline, newer failure taxonomy — labs run postmortems for model incidents exactly like infra ones.
