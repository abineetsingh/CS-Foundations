Security Review Runbook

A repeatable path from changed behavior to evidence-backed finding.

1. Scope the change

  1. What user or system capability changed?
  2. Which assets, identities, entry points, dependencies, and deployment settings are touched?
  3. What is explicitly outside this review?

2. State invariants

  • Who may perform each new action, on which object, under which conditions?
  • Which data must remain confidential, integral, available, fresh, or correctly attributed?
  • Which limits, ordering rules, and one-time transitions must survive concurrency and retries?

3. Trace authority

TraceAsk
Input → parserCan alternate encodings, duplicates, coercion, or size change interpretation?
Identity → policyWhere is caller identity established, and is permission checked for this action and object?
State → effectIs the transition atomic, idempotent, bounded, and safe under failure?
Dependency → privilegeCan an upstream response redirect, expand, select a file, or exercise this service's authority?
Error → response/logDoes failure expose data, preserve partial state, or fail open?

4. Prove the smallest case

Write one legitimate test and one abuse test. Change one variable. Prefer a harmless marker over data extraction, persistence, or disruption. A scanner result is a hypothesis until tied to code, violated policy, and affected asset.

5. Review the repair

Fix the root decision at the authoritative boundary. Search sibling paths. Preserve a regression test. Add containment and detection only where they address independent failure modes.

Finding format

Policy: [what must remain true]
Affected asset: [specific data/capability]
Preconditions: [attacker identity/access]
Evidence: [minimal reproducible behavior]
Root cause: [missing/incorrect decision]
Impact: [credible consequence and scope]
Repair: [root control + containment]
Verification: [denied abuse + allowed use + variants]
