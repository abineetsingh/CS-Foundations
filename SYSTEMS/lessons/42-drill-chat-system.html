Drill: Chat System

WhatsApp-lite under the timer: realtime delivery, message ordering, offline catch-up, group fan-out — phase 3's greatest hits wearing a product.

Why this matters

Chat is the standard mid-loop question: it forces WebSockets (29), wide-column storage (17), delivery guarantees (25), and fan-out — with no way to fake it synchronously. 45 minutes, timer, aloud, stages in order (40).

The prompt

Design a messaging app: 1:1 and group chats (≤200 members), delivery when online, catch-up when offline, sent/delivered/read indicators. 50 M DAU, each sending 40 messages/day. Message ≤ 1 KB.

Stage 1+2 — Requirements & envelope (10 min)

Compare your answer

Functional: send/receive 1:1 and group; offline catch-up; indicators. Descope aloud: media (presigned-URL pattern exists — 19), E2E encryption ("changes storage design fundamentally — assume TLS-only unless you want E2E"). Non-functional: delivery latency <500 ms online; no message loss, ever (durability outranks latency); per-conversation ordering (global ordering unnecessary — say so, it's a real insight); availability over consistency for presence/indicators (18 classification). Envelope: 50M × 40 = 2×10⁹ msgs/day ≈ 20k send/s (peak ~100k) — and each message delivers to ≥1 recipient: fan-out multiplies. Storage: 2 TB/day raw ≈ 700 TB/yr ×3 — sharded from day one, no deferring this time (contrast with 41 — the envelope DECIDED, both times: that's the method working).

Stage 3 — API & data model (5 min)

Compare your answer 
WebSocket: send{conv_id, client_msg_id, text} · recv{…} · ack{…}     (29)
REST:      GET /convs/{id}/messages?after_seq=N     ← the catch-up path
messages: (conv_id PARTITION KEY, seq SORT KEY, sender, text, ts)   → wide-column (17!)
convs, members, receipts(conv_id, user_id, last_delivered_seq, last_read_seq)

Two load-bearing choices: wide-column keyed by (conv_id, seq) — the textbook access pattern ("this conversation, in order") from 17, write-heavy at 20k/s which is Cassandra's home; and client_msg_id — the idempotency key (22) that makes send-retries safe over flaky mobile networks. Indicators as high-water marks (last_read_seq) not per-message rows — 200 members × per-message receipts would 200× your write load.

Stage 4 — High-level design (10 min)

Compare your answer

The 29 shape, verbatim: clients hold WebSockets to a gateway tier; registry (user→gateway) in Redis; stateless chat service behind it; message store (wide-column); pub/sub bus routing deliveries to the right gateways. Send path: gateway → chat svc → persist FIRST (assign seq, dedupe by client_msg_id) → ack sender → publish → recipient gateways push. Offline path: recipient reconnects → GET after_seq=N — push for speed, pull for truth (29's closing line, now your architecture).

Stage 5 — Deep dives (15 min)

Attempt first: (a) where does per-conversation ordering actually come from? (b) group fan-out: 200 members, one message — walk the multiplication; what about a 10,000-member channel? (c) exactly-once appearance to users: trace the guarantee end to end.

Compare: (a) ordering

The seq must be assigned at one serialization point per conversation — the storage partition (or a per-conv counter) is that point; conv_id partitioning (16/24: order-within-partition) makes it scale. Clients render by seq, and gaps trigger a pull — so even out-of-order delivery can't produce out-of-order display. Wall-clock timestamps are explicitly not the answer (clocks skew — 22's world has no shared clock).

Compare: (b) fan-out

200 members: persist once, then 200 registry lookups → pushes to ~online members; offline ones catch up by pull — so the write amplification is bounded by online membership. At 10,000+ (a "channel"), per-member push breaks: switch to pull-dominant — notify lazily, fetch on open (the fan-out read/write frontier — full treatment next lesson, and saying "this is the same problem as Twitter's celebrity" is exactly the cross-connection interviewers love).

Compare: (c) effectively-once to the eye

Mobile network drops the send-ack → client retries with the same client_msg_id → server dedupes (25's recipe: the (conv_id, client_msg_id) check is transactional with the insert). Delivery side: push may duplicate (redelivery after reconnect) → client dedupes by (conv_id, seq) — trivially, since seq is the render key. At-least-once everywhere, idempotence at both edges, user sees each message once. Phase 3, end to end, in one product feature.
