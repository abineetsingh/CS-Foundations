Topological Sort: Ordering by Dependency

Today's skill: order the nodes of a directed graph so every edge points forward — and detect when no such order exists (a cycle).

Why this, now

"Course Schedule" — can you finish courses given prerequisites? — is a top-tier interview question, and it's topological sort verbatim. For your research ambitions this is the big one: autograd executes computation graphs in topological order. Understanding this lesson is understanding how PyTorch schedules your model.

The problem

A directed edge u → v means "u must come before v" (prerequisite, dependency, import). A topological order lists all nodes so every edge points left-to-right. Two facts anchor everything:

  • An order exists if and only if the graph has no directed cycle (a cycle means "a before b before … before a" — absurd). Cycle-free directed graphs are called DAGs.
  • Orders are generally not unique — any valid one is accepted.

Kahn's algorithm: peel the free nodes

The idea a child could invent: repeatedly do whatever has no unmet prerequisites. Track each node's in-degree (incoming edge count); zero means "free to schedule now":

from collections import deque, defaultdict

def topo_sort(n, edges):                    # edges: (u, v) = u before v
    adj = defaultdict(list)
    in_deg = [0] * n
    for u, v in edges:
        adj[u].append(v)
        in_deg[v] += 1

    q = deque(i for i in range(n) if in_deg[i] == 0)   # free right now
    order = []
    while q:
        node = q.popleft()
        order.append(node)                  # schedule it
        for nb in adj[node]:
            in_deg[nb] -= 1                 # its obligation to nb is met
            if in_deg[nb] == 0:             # nb just became free
                q.append(nb)

    return order if len(order) == n else []   # leftovers ⇔ a cycle

O(V + E) — build, then each node enqueued once, each edge decremented once. The last line is the elegant part: nodes trapped in a cycle never reach in-degree 0, so they never enter order. Counting the output doubles as cycle detection — "Course Schedule" (can you finish?) is exactly len(order) == n, and "Course Schedule II" (give an order) is order itself.

The DFS alternative (post-order, reversed — every node emits after everything it points to) is worth knowing exists; Kahn's is easier to get right under pressure and its "process when in-degree hits zero" logic is how real schedulers work. One tool, well understood, beats two half-known.

Mission link · ML/AI

This is the closest an interview topic comes to your research goal. A model's forward pass must run ops in topological order (inputs before the ops consuming them); backprop runs the reverse topological order — that's the entire scheduling story of autograd. Same algorithm builds your software: make, package managers resolving install order, Airflow DAGs orchestrating training pipelines. "It's a DAG" is engineering shorthand you'll now never misparse.
