AI Systems II: Agents & RAG Infrastructure

One skill, learned today: design the layer above the model — retrieval pipelines and agent runtimes — as the distributed systems they secretly are.

Why this matters

Your mission's long-term clause — build agentic systems and ML infrastructure — lives in this lesson. It's also the frontier interview question at AI labs ("design a RAG system," "design a coding-agent platform"), where most candidates improvise. You won't: every component below is a lesson you've already passed, pointed at a new noun.

RAG: search (0020) grows a second index

Retrieval-augmented generation = fetch relevant documents, stuff them in the context, generate. Architecturally it is your 0020 derived-data search system with one addition — a vector index:

  • Ingest (write path): documents → chunk (~500-token pieces) → embed (an ML model maps each chunk to a vector where nearby = semantically similar) → store in a vector index. Fed asynchronously from truth via events/CDC — 0020/0024 verbatim, including the declared lag ("new docs searchable in ~seconds").
  • ANN, not exact: exact nearest-neighbor over 100 M vectors is a full scan (0013's sin). Approximate indexes — HNSW, a navigable graph — trade ~1–2% recall for O(log-ish) search: the B-tree bargain, re-signed for vectors (0013's tie-in, cashed).
  • Hybrid retrieval: BM25 (0020: exact words, names, codes) + vectors (meaning) with score fusion, then often a reranker. Production default; say "hybrid" before the interviewer does.
  • The eval loop is the real deep dive: retrieval quality (recall@k on a labeled set) is your SLI (0035); index/embedding changes ship behind eval-gated canaries (0034/0044). Everyone builds RAG; teams that measure retrieval keep theirs working.

The agent runtime: a distributed system with an LLM for a scheduler

An agent = a loop: model proposes a tool call → runtime executes it → result returns to context → repeat. Read that as an architect and the requirements write themselves — it's a workflow engine (0027) whose steps are chosen at runtime by a model:

RequirementIt is literally…
Tool calls fail/hang/duplicate0022's trio per call: timeouts, budgeted retries, idempotency keys on side-effecting tools
Multi-step work must survive crashesdurable execution: persist state after each step, resume from the log — the saga/outbox machinery (0027); this is what Temporal-style engines sell
Untrusted generated code/actionssandboxes (0032's containers as security walls) + least-privilege tool credentials (0037) + prompt-injection posture: tools ≥ untrusted-input handling
Runaway loops burning moneybudgets & kill switches — rate limiting (0028) in dollars and steps, not just tokens
Agent state & memorycontext = scratch (0004's split!); durable memory = a store; long transcripts = an event log per session (0024) — replayable for debugging
"What did the agent do for 30s?"tracing (0035): each tool call a span; the loop is a distributed trace
Many agents, queued work, priorities0023's queues + 0033's schedulers — an agent fleet IS a worker fleet

The interview close, worth memorizing: "An agent platform is a durable workflow engine with an LLM choosing the steps — so the hard problems are the classical ones: idempotent side effects, resumable state, sandboxing, budgets, and observability. The model is the easy part; the systems are the product." That sentence — which forty-four lessons earned you — is precisely what AI labs are hiring backend engineers to understand.
