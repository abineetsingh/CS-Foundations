Sorting: What Interviews Actually Require

Today's skill: wield Python's sort fluently (keys, stability), and know the landscape — what costs what, and when "sort first" unlocks a problem.

Why this, now

You will almost never implement a sort in an interview — you'll use one as a step. The real skills: custom keys without fumbling, knowing sort's O(n log n) price, and the strategic instinct for when paying it simplifies everything (it's the precondition for two pointers, binary search, and the entire intervals pattern).

Using Python's sort like a professional

arr.sort()                        # in place, returns None (a classic gotcha)
sorted(arr)                       # new list, original untouched
arr.sort(reverse=True)            # descending

words.sort(key=len)               # sort BY something: a key function
pairs.sort(key=lambda p: p[1])    # by second element
items.sort(key=lambda x: (x.priority, -x.score))   # tie-break via tuples

Both are Timsort: O(n log n) worst case, O(n) on already-mostly-sorted data, O(n) extra space, and stable — equal elements keep their original order. Stability is why the tuple-key trick works, and why you can sort by one field, then another, and both orders survive.

The landscape (concept level — implementation comes with recursion)

AlgorithmTimeSpaceWorth knowing because…
Merge sortO(n log n) alwaysO(n)the canonical divide & conquer — you'll build it in Lesson 18
QuicksortO(n log n) avg, O(n²) worstO(log n)partitioning idea powers quickselect (top-K without a heap)

Counting/bucket sortO(n + k)O(k)beats n log n when values live in a small range k — the loophole below

Two facts to say on cue: comparison-based sorting cannot beat O(n log n) (information-theoretic lower bound — n! orderings need log₂(n!) ≈ n log n comparisons to distinguish); and counting sort escapes the bound by not comparing at all — it indexes by value. "Sort Colors" (three values) is the classic exploit.

The strategic move: buy structure, then cash it in

Unsorted data supports almost no clever patterns. Paying O(n log n) up front buys sortedness, which unlocks:

  • Two pointers (Lesson 06) — pair-finding drops to O(n)
  • Binary search (Lesson 14) — lookups drop to O(log n)
  • Neighbors become adjacent — closest pair, duplicates, "merge intervals" (Lesson 39) all become single passes

Decision rule: if a hash map gives O(n) without sorting, prefer it. But if the problem talks about order, ranges, closeness, or intervals, "what if I sort first?" should be among your first three thoughts. One caveat: sorting destroys original indices — if the answer needs them, sort (value, index) pairs or use the hash map instead (Two Sum's trap).

Mission link · ML/AI

Every top-k sampling step sorts (or partial-sorts) the logits; retrieval systems sort candidates by similarity score; and "argsort" is a tensor primitive you'll meet constantly. The n log n lower bound is one reason vector databases work hard to avoid full sorts — approximate top-k beats exact ordering at scale (Lesson 23 shows the heap version).
