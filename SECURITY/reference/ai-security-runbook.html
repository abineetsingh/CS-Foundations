Prompt-injection review

  1. Separate user intent, developer policy, and untrusted content by provenance.
  2. Assume direct and indirect instructions can influence the model; delimiters are context, not a security boundary.
  3. Minimize accessible data and actions before inference.
  4. Authorize tool, arguments, resource, destination, budget, and identity outside the model.
  5. Bind human approval to the exact effect; invalidate it when arguments or context change.
  6. Measure unauthorized effects and disclosure—not whether the response contains a preferred refusal phrase.

Eval card

System version: model + prompt + harness + tools + policies
Threat case: actor, source, goal, severity, expected security outcome
Metrics: attack success rate, false-block rate, severity-weighted loss
Segments: direct/indirect, source, tool, tenant, model, language, encoding
Evidence: repeated trials, held-out and adaptive cases, confidence interval
Gate: threshold, owner, exception, residual risk, rollback trigger

Release gate

  • High-risk flows have owners, controls, and tests.
  • Retrieval authorization and tenant isolation fail closed.
  • Model output is parsed, validated, and authorized before effects.
  • Tool calls use narrow schemas, scoped identities, budgets, and meaningful approval.
  • Data maps cover prompts, providers, memory, embeddings, traces, feedback, and deletion.
  • Artifacts have identity, provenance, safe loading, immutable promotion, and rollback.
  • Privacy-aware traces reconstruct policy decisions without default raw-secret logging.
  • Security eval thresholds and utility regressions are explicit release decisions.
Core principle

The model is one fallible component. The application harness owns enforceable authorization, data boundaries, execution policy, evidence, and safe degradation.
