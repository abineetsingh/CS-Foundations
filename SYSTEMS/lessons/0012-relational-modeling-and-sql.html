Relational Modeling & SQL

One skill, learned today: turn a feature description into tables, keys, and relationships — and query them with joins.

Why this matters

Phase 2 opens the box you've been drawing as a single rectangle. "Design the schema" is a standard interview sub-question, and modeling is where real backends are won or lost — a bad schema haunts a company for a decade. The relational model is the default; everything else (lesson 0017) defines itself against it.

The model: tables, keys, relationships

A relational database holds tables of uniform rows. Two kinds of keys wire tables together: a primary key uniquely names each row; a foreign key is a column holding another table's primary key — a typed pointer. All relationships reduce to three shapes:

ShapeExampleModeled as
One-to-manya user has many tweetstweets.user_id → FK to users.id
Many-to-manyusers follow usersa join table: follows(follower_id, followee_id)
One-to-oneuser ↔ user_settingsFK with a uniqueness constraint (rare; often just more columns)

Hands-on: model it in SQLite

No new installs — the sqlite3 shell ships with macOS:

sqlite3 twitter.db
CREATE TABLE users  (id INTEGER PRIMARY KEY, name TEXT NOT NULL);
CREATE TABLE tweets (id INTEGER PRIMARY KEY, user_id INTEGER NOT NULL REFERENCES users(id),
                     text TEXT NOT NULL, created_at TEXT DEFAULT (datetime('now')));
CREATE TABLE follows (follower_id INTEGER REFERENCES users(id),
                      followee_id INTEGER REFERENCES users(id),
                      PRIMARY KEY (follower_id, followee_id));

INSERT INTO users (name) VALUES ('ada'), ('grace'), ('alan');
INSERT INTO tweets (user_id, text) VALUES (2, 'compilers!'), (3, 'machines think?'), (2, 'COBOL day');
INSERT INTO follows VALUES (1, 2), (1, 3);        -- ada follows grace and alan

Now the query that is the product — ada's home timeline. A JOIN matches rows across tables by key:

SELECT u.name, t.text, t.created_at
FROM follows f
JOIN tweets t ON t.user_id = f.followee_id     -- tweets by people ada follows
JOIN users  u ON u.id = t.user_id              -- attach author names
WHERE f.follower_id = 1
ORDER BY t.created_at DESC
LIMIT 20;

Read it aloud: start from ada's follow list, join to those users' tweets, join again for names, newest first, page of 20. Being able to speak a join like a sentence is the skill [SQLite: SELECT].

Normalize by default, denormalize by evidence

Normalization = every fact stored once. Ada's name lives in one row; a rename is one UPDATE, and no copy can disagree. Denormalization = deliberately duplicating (e.g., storing author_name inside each tweet row) to skip a join on a hot path — buying read speed with update pain and drift risk. The discipline: normalize first; denormalize only when a measured hot path demands it, and say so out loud in interviews — it's a tradeoff declaration, like cache staleness in lesson 0009 [DDIA ch. 2].

AI-lab tie-in

Feature stores and training-data pipelines are schema design under another name — and "text-to-SQL" is a standard LLM eval task. An agent granted database tools is only as safe and effective as the schema is legible: clear names, real constraints, documented keys. Modeling skill transfers straight into agent-tool design.
