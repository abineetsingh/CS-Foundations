Replication

One skill, learned today: add replicas to a database — for survival and for read scale — and name the lag-shaped bugs that come free with them.

Why this matters

Since lesson 0005 you've carried an open wound: notes.db lives on ONE disk. Replication closes it — and simultaneously answers "how do we scale reads past the cache?" It also introduces the course's first genuinely distributed problem: copies that disagree. Phase 3 grows out of this lesson.

Leader–follower: the standard shape

Replication = keeping live copies of the same data on multiple machines. The dominant arrangement is leader–follower (primary/replica): all writes go to the one leader; the leader streams its change log to followers, which apply it in order; reads can go to any copy [DDIA ch. 6]. This buys three things at once:

  • Durability beyond one disk — the machine can burn down; the data exists elsewhere.
  • Read scale — 3 replicas ≈ 3× read throughput, stacking with your cache (0009): misses spread over followers.
  • Availability — leader dies → promote a follower (failover). Sixty seconds of automated drama instead of a day of data recovery.

Why one leader? Because single-writer sidesteps write conflicts entirely — two simultaneous writes meet at one place, where transactions (0014) already know how to referee them. Multi-leader exists (multi-region) and buys those conflicts back; treat it as an advanced tool you name, not reach for.

The catch: replication lag

The leader doesn't wait for followers (asynchronous replication — waiting on every follower would make every commit as slow as the slowest machine, and a dead follower would freeze all writes). AI-lab tie-in

Model weights are replicated state with a luxurious property: they're immutable between releases — replication with zero lag problems (that's just a file copy). The actual replication headaches at AI labs live in the ordinary places: user data, conversation history, usage/billing counters.
