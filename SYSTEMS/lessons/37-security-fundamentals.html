Security Fundamentals

One skill, learned today: place the security layers every backend needs — authentication, authorization, TLS, secrets — and name the habits that block the common attacks.

Why this matters

Interviews increasingly end designs with "how is this secured?", and a five-sentence fluent answer is the difference between "engineer" and "liability." This is defensive fundamentals — the hygiene layer every production system owes its users — not offense; depth beyond this lives with security teams.

Authentication vs authorization — the 401/403 divide, industrialized

You met the distinction in lesson 02; now the machinery. Authentication ("who are you?") for APIs is usually one of: session cookies (server-held session in Redis — your 07 externalized state — referenced by an opaque cookie) or signed tokens (JWTs): the server signs a blob of claims (user_id, expiry, roles); any server can verify the signature without shared state — statelessness for identity [MDN: HTTP auth]. The tradeoff to recite: tokens can't be revoked before expiry (they're valid by math, not by lookup) — so keep them short-lived and pair with refresh tokens; sessions revoke instantly but cost a store lookup. OAuth2/OIDC is the delegation protocol behind "Sign in with Google" — know its shape (redirect → consent → code → token), not its RFCs.

Authorization ("are you allowed?") is where real-world breaches concentrate — especially object-level checks: GET /notes/17 must verify note 17 belongs to this caller, not just that the caller is logged in. Skipping that check (guessing ids: 17, 18, 19…) is IDOR — #1 in the OWASP API Top 10 [OWASP API Top 10]. The habit: authorize every object access, in one shared code path, close to the data.

The transport and the perimeter

The layers in the flow, compressed: TLS everywhere (the S in HTTPS — encryption + server identity via certificates; terminate at the LB/gateway per 08, and modern zero-trust shops encrypt service-to-service too with mTLS). The gateway (30) authenticates once, rate-limits (28), and strips malice before requests reach services. Validate at the boundary — Pydantic (03) plus parameterized SQL (04) already made you injection-resistant; the generalized creed is never interpolate untrusted input into anything that gets executed (SQL, shell, HTML). Secrets (DB passwords, API keys) never live in code or images (32 images get pulled and inspected!) — they come from a secrets manager (Vault, cloud KMS) injected at runtime, rotated regularly. Least privilege: each service's credentials can touch only what it needs — so a compromised notes service can't read the payments database (bulkheads, 36, for permissions).

AI-lab tie-in

LLM systems add a genuinely new boundary problem: prompt injection — untrusted content (a webpage an agent reads) smuggling instructions into the model's input. It's the injection family (never mix untrusted data with trusted instructions) but without a clean escaping fix, which is why agent security leans on the old tools: least-privilege tool permissions, sandboxed execution (32's containers), human confirmation for irreversible actions. Your API-security instincts transfer straight into agent design — and AI labs interview on exactly this bridge.
