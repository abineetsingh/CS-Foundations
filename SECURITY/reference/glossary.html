Security Engineering Glossary

Course vocabulary, kept precise and stack-agnostic. This document grows as terms enter the curriculum.

Foundations

Asset
Something whose confidentiality, integrity, availability, authenticity, or correct use matters: data, money, identities, compute, reputation, or a business capability.
Actor
A person, service, device, process, or organization that interacts with the system. Actors have capabilities and goals; “user” does not imply trusted.
Adversary
An actor deliberately trying to violate the intended security policy. Model concrete capabilities and access, not a vague all-powerful attacker.
Security policy
A falsifiable statement describing which actor may perform which action on which asset under which conditions.
Trust
A dependency on an actor or component behaving as expected. Trust is an assumption to justify and minimize, not a feeling of safety.
Trust boundary
A place where data or control crosses between components with different identities, owners, privilege, or assurance. Crossing it demands explicit validation and authorization.
Attack surface
The reachable interfaces, inputs, workflows, dependencies, identities, and operational paths through which an adversary may affect assets.
Vulnerability
A weakness that permits some threat to violate the intended policy under stated conditions.
Threat
A plausible event or action that could harm an asset. A threat model connects actors, capabilities, paths, assets, and mitigations.
Security control
A design, implementation, configuration, or process intended to prevent, detect, limit, or recover from a policy violation.
Mitigation
A response that reduces a threat's likelihood or impact. Good mitigations state assumptions and have a verification method.
Security invariant
A property that must remain true across every valid state transition, such as “a tenant can read only records scoped to that tenant.”
Least privilege
Grant an identity only the actions and resources it needs, under the narrowest practical conditions and duration.
Defense in depth
Independent layers that reduce the chance one control failure becomes complete compromise. More controls are useful only when they address distinct failure modes.
Verification
Evidence that a control meets a stated requirement under legitimate and adversarial conditions. A security claim without a test remains an assumption.

Application vulnerabilities

Injection
Untrusted data changes the syntax or behavior of an interpreter such as SQL, a shell, or a template engine.
SQL injection
Input changes SQL query structure instead of remaining a bound value.
SSRF
Server-side request forgery: an attacker influences a server to connect to an unintended destination using the server's network position or credentials.
Path traversal
Attacker-controlled file references escape an intended filesystem root through path semantics, links, encodings, or archive entries.
Unsafe deserialization
Untrusted serialized data selects types or reconstruction behavior with security-relevant side effects.
XSS
Cross-site scripting: untrusted data becomes executable browser syntax in an HTML, attribute, URL, JavaScript, CSS, or DOM context.
CSRF
Cross-site request forgery: another site causes a browser to exercise ambient user credentials on a target application.
CORS
Browser protocol for allowing selected origins to read cross-origin responses; it is not authentication and does not generally stop requests from being sent.
Business-logic vulnerability
A sequence of individually valid actions violates an application-specific rule or invariant.
Race condition
Security behavior depends on timing or interleaving between operations, often violating an invariant between check and effect.

Identity and access

Authentication
Establishing which identity is interacting, with a stated assurance level.
Authorization
Deciding whether a specific identity may perform a specific action on a specific resource under current conditions.
BOLA / IDOR
Broken object-level authorization: a caller can act on an object because the server fails to bind identity to permission over that object.
Session
Server-recognized continuity that binds requests to prior authentication state, commonly represented by a bearer identifier.
MFA
Authentication using more than one factor category; factors differ substantially in phishing and replay resistance.
OAuth 2.0
A framework for delegated authorization in which a client obtains scoped access to a resource server.
OpenID Connect
An identity layer over OAuth 2.0 that supplies authenticated claims about an end user.
JWT
A compact claim container. Signed JWTs provide integrity and origin authentication, not secrecy; safe use requires full claim and key validation.
RBAC
Role-based access control: permissions are grouped into roles assigned to identities.
ABAC
Attribute-based access control: policy evaluates attributes of subject, resource, action, and environment.

Secure delivery and cryptography

SAST
Static application security testing: automated analysis of source or compiled code without exercising a deployed application.
DAST
Dynamic application security testing: automated interaction with a running application from the outside.
SCA
Software composition analysis: inventory and known-risk analysis for dependencies and components.
SBOM
Software bill of materials: an inventory of components included in a software artifact.
CSPRNG
Cryptographically secure pseudorandom number generator suitable for keys, tokens, nonces, and other unpredictable values.
MAC
Message authentication code: a keyed construction that provides integrity and authenticity to parties sharing the key.
AEAD
Authenticated encryption with associated data: encryption that provides confidentiality and integrity while binding optional unencrypted context.
Nonce
A value used once under a given cryptographic key according to the construction's requirements; it often need not be secret.
Digital signature
A private-key operation whose result can be verified with the corresponding public key to establish message origin and integrity.
TLS
Protocol providing an authenticated, encrypted transport channel; its protection exists between actual termination points.

Cloud and platform security

IAM
Identity and access management for principals, credentials, policies, roles, trust relationships, and resource permissions.
Temporary credentials
Short-lived credentials issued dynamically to an identity or workload, reducing exposure compared with embedded long-lived keys.
Workload identity
A platform-issued identity representing a running service or workload rather than a human or embedded shared secret.
Control plane
Administrative interfaces that configure, create, modify, and govern infrastructure or services.
Data plane
Interfaces through which a deployed service processes its ordinary application traffic and data.
Container
A process-isolation and packaging mechanism that normally shares the host kernel; it is not a virtual-machine boundary by default.
Kubernetes RBAC
Cluster authorization over API verbs and resources, commonly granted to users, groups, and service accounts through roles and bindings.
Pod Security Standards
Kubernetes profiles—Privileged, Baseline, and Restricted—that constrain security-relevant pod configuration.

AI and LLM security

Large language model (LLM)
A probabilistic model that generates or scores sequences. Its output is untrusted application input, not an authorization or factual oracle.
Agent
A system combining a model with a harness, context, memory, tools, identities, and an environment to pursue goals across multiple steps.
Agent harness
Ordinary application code that assembles context, routes models, exposes tools, applies policy, records state, and controls execution.
Prompt injection
Attacker-influenced natural-language content attempts to change model behavior beyond the user's authorized intent. Direct injection comes from the user; indirect injection arrives through external content such as webpages, documents, messages, or tool results.
RAG
Retrieval-augmented generation: external information is retrieved and placed into model context. Relevance does not imply authorization, provenance, freshness, or trust.
Context poisoning
Adversarial content is placed into retrieved data, memory, caches, or tool results so it persistently influences later model behavior.
Excessive agency
An AI system has more actions, permissions, autonomy, destinations, or duration than its user-authorized task requires.
Guardrail
A deterministic policy or probabilistic classifier placed at an AI-system boundary to block, transform, or route risky input, context, tool calls, or output. A guardrail does not itself grant authorization.
Security eval
A versioned set of threat-derived cases, scorers, metrics, and release thresholds used to measure security behavior across system changes.
Attack success rate (ASR)
The fraction of adversarial trials that achieve the defined harmful outcome. The outcome and denominator must be explicit.
False-block rate
The fraction of legitimate trials incorrectly blocked by a security control; it measures utility cost alongside attack reduction.
Model or data poisoning
Manipulation of a model artifact, training data, fine-tuning data, or evaluation data to produce attacker-beneficial behavior.
LLM jacking
Unauthorized use of stolen or exposed AI service credentials or infrastructure, often causing compute cost, capacity loss, or policy abuse.
Model Context Protocol (MCP)
A protocol for connecting AI applications to tools and context providers. Transport authorization does not replace per-tool, per-resource application authorization.
