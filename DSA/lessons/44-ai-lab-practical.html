AI-Lab Practical Screens: Build a Working Thing

Today's skill: the practical coding round — a small, real system built to working order in 45–60 minutes, tests and all.

Why this, now — with an honesty note

Frontier labs lean toward practical screens: "build a rate limiter," "implement this spec," sometimes with your editor and even AI tools available. Candidate reports (see r/leetcode threads and interview-experience posts) consistently describe this format for OpenAI and Anthropic — but formats change; re-research the specific loop 2–3 weeks before applying (this is the open Gap in RESOURCES.md). What won't change: the skills below.

The protocol adjustment

Lesson 41's loop applies with two amendments:

  • Skeleton first, then flesh: write the class with stubbed methods and a working end-to-end call in the first 10 minutes — then deepen. A thin working slice beats a deep broken one; it also absorbs spec-changes gracefully.
  • Test as you go: after each method, three lines in __main__ proving it. Narrate: "let me verify before building on this." That habit is precisely what they're screening for.

Worked example: a sliding-window rate limiter

"Allow at most k requests per client per w seconds." Note how much of this course composes here — deque (11), sliding windows (08), hash maps (05), design composition (43):

import time
from collections import defaultdict, deque

class RateLimiter:
    def __init__(self, max_requests, window_seconds):
        self.k = max_requests
        self.w = window_seconds
        self.log = defaultdict(deque)          # client → timestamps in window

    def allow(self, client_id, now=None):      # injectable clock: testable!
        now = time.monotonic() if now is None else now
        q = self.log[client_id]
        while q and q[0] <= now - self.w:      # evict expired (variable window!)
            q.popleft()
        if len(q) < self.k:
            q.append(now)
            return True
        return False

if __name__ == "__main__":                     # tests-as-you-go, demo style
    rl = RateLimiter(max_requests=2, window_seconds=10)
    assert rl.allow("a", now=0.0) is True
    assert rl.allow("a", now=1.0) is True
    assert rl.allow("a", now=2.0) is False     # k exhausted
    assert rl.allow("b", now=2.0) is True      # clients independent
    assert rl.allow("a", now=11.0) is True     # window slid past t=0
    print("all good")

Interview-grade touches to narrate: the injectable clock (deterministic tests — no sleep()); amortized O(1) per call (each timestamp enqueued/evicted once — Lesson 08's argument); memory O(k) per active client; and the trade-off menu if asked to scale (token bucket for O(1) memory, fixed-window counters for approximate limits). Likely curveballs: per-endpoint limits (key by tuple), a retry_after() method (q[0] + w − now), thread safety (name the lock, don't build it).

The practice regimen (this is the lesson)

One per sitting, 45-minute timer, tests included, out loud:

  1. Rate limiter (above — build it yourself first, then compare)
  2. In-memory KV store with TTL — get/set/delete + expiry; curveball: keys(prefix) (hello, Lesson 25)
  3. LRU cache (43) — now under the clock, with tests
  4. Log/event aggregator — ingest (timestamp, event), query counts in a range (09's prefix thinking + 14's bisect)
  5. Greedy longest-match tokenizer — given a vocab, split text into tokens, longest match first (25; the ML-flavored classic)

These five aren't random: rate limiters front every model API; KV stores with TTL are feature caches; LRU is the inference cache (43); log aggregation is eval/telemetry plumbing; and the tokenizer is the actual front door of an LLM. Passing this round and being useful in week one at a lab are, for once, the same skill.
