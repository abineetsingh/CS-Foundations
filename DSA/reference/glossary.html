Glossary

The course's nomenclature, one line each. Terms are used consistently in every lesson; if a definition here surprises you, revisit its lesson.

TermMeaningLesson
Big-OHow work grows as input grows; constants and lower-order terms dropped.01
auxiliary spaceExtra memory beyond the input; what "space complexity" means by default.02
in placeO(1) auxiliary space; the input is mutated rather than copied.02
amortizedCost averaged over a sequence of operations; rare expensive steps spread thin (append's resize).03
dynamic arrayArray with spare capacity that doubles when full — Python's list.03
immutable / hashableCan't change after creation / has a stable hash — immutability is why strings and tuples can be dict keys.04–5
hash tableArray indexed by hash(key); O(1) average lookup/insert/delete. Python's dict and set.05
collisionTwo keys landing in one slot; handled internally, cause of the "average" caveat.05
invariantA property kept true at every step; the tool for writing correct loops (binary search, windows, BSTs).08, 14
monotonic stack/dequeStack/deque kept sorted by evicting on arrival; solves next-greater and window-max in O(n).10–11
dummy nodeFake node planted before a linked list's head so head-changes aren't special cases.12
fast/slow pointersTwo pointers at different speeds; finds middles and detects cycles (Floyd).13
answer-space searchBinary search over candidate answers using a monotonic yes/no feasibility check.15
stable sortEqual elements keep their original order (Timsort is; enables multi-key sorting).16
base case / leap of faithRecursion's floor / trusting recursive calls on smaller inputs — induction, operationalized.17
divide & conquerSplit, solve halves by trust, combine; priced via work-per-level × levels.18
pre/in/post-orderDFS variants by when the node is handled: before / between / after its subtrees.19
level-order (BFS)Queue-driven traversal by distance; the level-size trick processes levels as units.20
BST invariantLeft subtree < node < right subtree, at every node, over WHOLE subtrees.21
O(h)Tree-op cost in height: log n balanced, n degenerate — always state both.21
heapComplete tree in an array; parent ≤ children; O(1) peek-min, O(log n) push/pop.23
top-K patternMin-heap of size k holding the current elite; O(n log k).23
trieTree whose paths spell prefixes; O(L) ops; the structure for "starts with".25
backtrackingDFS over a decision tree with choose → explore → un-choose state restoration.26
pruningRejecting a partial path early, discarding its whole subtree unexplored.27
adjacency listnode → list of neighbors; O(V+E) space; the default graph representation.28
visited setWhat makes graph traversal terminate; permanent (traversal) vs path-scoped (backtracking).28–29
DAGDirected acyclic graph — exactly the graphs that can be topologically ordered.30
topological orderNode ordering where every edge points forward; Kahn's = repeatedly take in-degree-0 nodes.30
union-find (DSU)Forest tracking groups under merges; path compression + rank → O(α) ≈ constant.31
relaxation / finalizationImproving a distance estimate / Dijkstra's pop-time commitment (safe iff weights ≥ 0).32
MSTCheapest edge set connecting all nodes; Kruskal = sort + union-find; cut argument proves greed.33
overlapping subproblemsThe same subproblem recurring — the property that makes caching profitable; DP's diagnostic.34
memoization / tabulationTop-down cache on recursion / bottom-up table filled in dependency order.34–35
state / transitionWhat identifies a subproblem / the menu of last moves relating states — DP questions 1 & 2.34
rolling variablesCollapsing a DP table to O(1) when the transition looks back a bounded distance.35
pseudo-polynomialPolynomial in a numeric VALUE (like capacity W), not the input's size — knapsack's fine print.37
exchange argumentGreedy proof: any optimal solution can be morphed into the greedy one at no cost.38
sweep lineWalk sorted event points (+1 start, −1 end) tracking a running level; intervals' engine.39
bitmaskInteger used as a set of booleans; subsets of n items ↔ integers 0..2ⁿ−1.40
the six stepsClarify → example → brute force → optimize → code → verify; the interview protocol.41
composition (design)Pairing structures so each covers the other's weak op, kept in sync (map + linked list = LRU).43
