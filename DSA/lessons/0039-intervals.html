Intervals: Sort, Then Sweep

Today's skill: the interval playbook — sort by start (or end), sweep once, and reason about exactly one thing: overlap.

Why this, now

Intervals are a small, closed world: five or so canonical problems cover nearly every interview appearance, all solved by one sorted sweep. Calendar systems, resource booking, and rate limiting are intervals in production — this pattern earns its keep beyond interviews faster than most.

The one geometric fact

a = [1, 5]   b = [4, 8]     overlap ⇔ a.start ≤ b.end AND b.start ≤ a.end
      ├────a────┤
           ├──────b───┤     equivalently, NOT (a ends before b starts
                                              OR b ends before a starts)

Write this check without thinking. Off-by-one alert: decide early whether touching intervals ([1,5], [5,8]) count as overlapping — problems differ, and the ≤ vs < follows from that decision.

Canonical problem 1: Merge Intervals

Coalesce all overlapping intervals. Sorting by start (Lesson 0016's "buy structure") makes overlap purely a neighbor affair — each interval can only merge with the one growing block before it:

def merge(intervals):
    intervals.sort(key=lambda iv: iv[0])          # by start
    result = [intervals[0]]
    for start, end in intervals[1:]:
        last = result[-1]
        if start <= last[1]:                      # overlaps the growing block:
            last[1] = max(last[1], end)           #   extend it (max: containment!)
        else:
            result.append([start, end])           # gap: start a new block
    return result

O(n log n) for the sort, O(n) sweep. The max guards against contained intervals ([1,10] then [2,3] — the block's end must not shrink). Without the sort, every pair must be compared: O(n²) and fiddly; with it, one look-back suffices. That's the entire playbook: sort so that only adjacent things interact, then sweep with O(1) state.

Canonical problem 2: Meeting Rooms II (how many rooms?)

Minimum rooms so no two overlapping meetings share one. Insight: the answer is the maximum number of meetings simultaneously in progress. Split starts and ends into two sorted streams and sweep time:

def min_rooms(intervals):
    starts = sorted(iv[0] for iv in intervals)
    ends   = sorted(iv[1] for iv in intervals)
    rooms = best = 0
    s = e = 0
    while s < len(starts):
        if starts[s] < ends[e]:      # a meeting starts before the earliest end:
            rooms += 1               #   +1 concurrent
            s += 1
        else:                        # earliest-ending meeting finishes first:
            rooms -= 1               #   a room frees up
            e += 1
        best = max(best, rooms)
    return best

Two pointers (Lesson 0006) over two sorted streams — a k=2 merge (Lesson 0024)! The alternative implementation keeps a min-heap of end times (push each meeting, pop all that ended); same O(n log n), and naming both is a flex. This sweep-line idea — walk time, +1 on start, −1 on end, track the running level — generalizes to skyline problems, CPU load, concurrent-users questions.

Mission link · ML/AI

GPU cluster schedulers solve meeting-rooms daily: jobs are intervals over devices, and max-concurrency = required capacity. Sliding-window rate limiters (Lesson 0044 builds one!) count requests in a time interval; audio/video ML pipelines merge detection intervals ("speech from 0:03–0:07") exactly like merge().
