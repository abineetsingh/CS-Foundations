The 45-Minute Interview Algorithm

One skill, learned today: run a system design interview on rails — five stages, timed, with you driving — so knowledge you already have arrives in the order interviewers score.

Why this matters

This is the near-term mission checkpoint itself. You now know more than enough content; interviews are lost on process — rabbit-holing, waiting to be led, never stating tradeoffs. The algorithm below is the standard structure hiring managers themselves teach [Hello Interview: delivery framework]. Drills 41–43 run it against real prompts.

The five stages (with the clock)

StageTimeWhat you produceCourse tools
1. Requirements~5 minfunctional list (3–5 features, aggressively descoped: "I'll focus on X and Y; skipping login — fair?") + non-functional list (scale, latency targets, consistency needs, read:write ratio)11's functional/non-functional split
2. Envelope~5 minQPS (read AND write), storage, the ratio — and a conclusion: "reads are the problem, so: cache-heavy"06, verdict-first
3. API + data model~5 mincore endpoints (REST, paginated) + main tables/stores with keys02, 12, 17
4. High-level design~10 minthe boxes, drawn while narrating the read path and write path end to end — a working v1, no cleverness yet11's diagram + queues (23) where async
5. Deep dives~15 minattack the 2–3 hardest problems (the interviewer's real interest): the hot key, the fan-out, the consistency call, the failure mode — with tradeoffs stated per optionphases 2–4, all of them

The meta-skills that score

  • Drive. Announce the structure ("let me get requirements, size it, then design — deep dives at the end"), narrate transitions, and propose the deep dives yourself. Interviewers steer when needed; silence while you scribble is the anti-pattern.
  • Tradeoffs out loud, always. Never "I'd use Cassandra" — always "wide-column here because the access pattern is append-heavy time-series (17); Postgres would need sharding by month and loses us nothing… actually let me note both and pick per the write QPS." The form "X because Y; the alternative costs Z" is what gets transcribed into feedback as 'strong signal'.
  • Declare your budgets. Staleness (09/18), latency (05), error budget (35) — numbers you commit to, unprompted. It's the single strongest experience-signal available in 45 minutes.
  • Meet pushback with curiosity. "What if a celebrity posts?" is a gift — the interviewer is pointing at the intended deep dive. Say "great — that breaks my per-user cache assumption; here are two mitigations…" (16). Defensiveness fails candidates whose designs were fine.
  • Know your seniority dial. Junior loops want a clean stage 4. Senior loops live in stage 5 — go there faster, flag the hard parts yourself ("the interesting problem here is the fan-out; want me to start there?").

Print the interview runbook — the one-page version of this lesson — and keep it beside you during every drill from here on.

AI-lab tie-in

AI-lab loops run this same algorithm with two twists: prompts skew toward their world ("design an LLM inference API," "design evals infrastructure" — lessons 44–45 arm you), and deep dives skew toward scale + cost ("GPU utilization is 40% — what do you change?"). The structure you drill now is the structure you'll use there verbatim.
