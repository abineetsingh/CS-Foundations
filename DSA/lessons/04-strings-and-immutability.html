Strings & Immutability

Today's skill: handle strings at the right cost — knowing that every "modification" of a Python string is secretly a full copy.

Why this, now

A huge share of interview problems are string problems (palindromes, anagrams, substrings). One immutability mistake turns an O(n) answer into O(n²) without changing a single visible behavior — the interviewer will ask you to spot it.

Strings are immutable arrays

A Python string is a contiguous array of characters (Lesson 03 applies: s[i] is O(1), len(s) is O(1), slicing copies). The twist: it cannot be changed after creation.

s = "hello"
s[0] = "H"        # TypeError — no mutation, ever
s = "H" + s[1:]   # legal, but builds a NEW 5-char string: O(n)

So every operation that "changes" a string — concatenation, replace, upper — allocates a new one and copies. There is no in-place edit.

The classic trap: concatenation in a loop

def join_words_slow(words):     # n words
    sentence = ""
    for w in words:
        sentence = sentence + w   # copies EVERYTHING so far
    return sentence

Each + copies the whole accumulated string: 1 + 2 + 3 + … + n copies ≈ n²/2. That's O(n²) — the same all-pairs growth as a nested loop, hiding in one innocent line. The fix:

def join_words(words):
    return "".join(words)         # one pass, one allocation: O(n)

Or, when building character by character: append to a list (amortized O(1) per append — Lesson 03), then "".join(parts) once at the end.

The working idioms

TaskIdiomCost
Build a string incrementallylist of parts + "".join()O(n)
Edit characterschars = list(s) → mutate → "".join(chars)O(n)
Reverses[::-1]O(n)
Char → number, number → charord("a") = 97, chr(97) = "a"O(1)
Compare letter counts (anagrams)count into a dict — Lesson 05O(n)

One more consequence of immutability worth saying in interviews: strings are hashable, so they can be dict keys and set members — which is why word-counting with a dict works at all (next lesson).

Mission link · ML/AI

LLMs barely touch strings. A tokenizer converts text to integer IDs precisely because integers in contiguous arrays are cheap and strings are not — every downstream op becomes array math. When you later meet BPE tokenization, you'll recognize it as repeated string-pair merging done carefully to avoid exactly the O(n²) trap above.
