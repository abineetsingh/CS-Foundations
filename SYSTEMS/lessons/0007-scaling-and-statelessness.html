Scaling & Statelessness

One skill, learned today: explain the two ways to scale, why the second one wins, and the one property your servers must have to allow it.

Why this matters

Lesson 0006's envelope said "50,000 peak QPS" — one machine can't serve that. What you do next is the fork in every architecture: scale up or scale out. Interviews assume you'll scale out and probe whether you understand the price of admission: statelessness.

Up vs out

Vertical ("scale up")Horizontal ("scale out")
Movebuy a bigger machineadd more machines
Effortzero code changes — just moneyarchitecture must cooperate
Ceilinghard: the biggest machine money buys, and prices grow superlinearlysoft: keep adding (Google-scale runs this way)
Failure storythe one machine dies → you're downone of N dies → capacity dips ~1/N

Verdict used in practice: scale up first (it's cheap in effort — never build a distributed system you don't need), but design so that scaling out stays possible — because only out removes both the ceiling and the single point of failure. The mental shift is pets → cattle: servers stop being precious named individuals and become an interchangeable herd.

The price of admission: statelessness

Run two copies of your lesson-0003 notes API (the dict version) and route users to either copy: notes created on server A don't exist on server B. Worse with logins — if server A holds "user 42 is logged in" in memory, requests landing on B get 401s. Requests must be free to land anywhere, so:

The rule

A stateless service keeps no client-visible state between requests in its own memory. All truth lives in shared external stores (database, cache, object storage). Any server can then serve any request — which is exactly what lets you add servers freely [Twelve-Factor App: processes].

The classic offender is the session — "who is logged in." The two production fixes: externalize it (session data in a shared store like Redis, keyed by a cookie) or carry it in the request (a cryptographically signed token the server can verify without remembering anything — the JWT idea; details in lesson 0037). "Sticky sessions" (pin each user to one server) is the crutch to name-and-avoid in interviews: it breaks the moment that server dies.

AI-lab tie-in

LLM inference fleets are the same shape: any replica can serve any prompt because the weights are identical and the conversation state arrives in the request (the context) or sits in an external KV-cache tier. When you hear "we route long conversations back to the same replica to reuse its KV-cache" — that's sticky sessions, same tradeoff, same failure mode.
