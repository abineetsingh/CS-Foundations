Queues & Async Work

One skill, learned today: take slow work out of the request path — answer now, work later, survive burst and failure via a queue.

Why this matters

Lesson 11 flagged v1's second honest gap: everything synchronous — the client waits for every step. The message queue is the box that fixes it, and it's the most common "how would you improve this?" answer in interviews: video encoding, email sending, feed fan-out, image thumbnails — anything slow, bursty, or failable belongs behind one.

The move: decouple asking from doing

Uploading a video shouldn't hold the user hostage for the 10 minutes it takes to transcode. Split the work: the request path does the minimum (record intent, enqueue a job, return 202 Accepted); workers — a separate fleet of processes — consume jobs at their own pace. The queue between them (SQS, RabbitMQ, Redis lists; your DSA 11 deque, industrialized) is a durable buffer that grants four powers [AWS: message queues]:

PowerMeaning
Latencythe user waits for the enqueue (~ms), not the work (~minutes)
Burst absorption10,000 uploads in one minute → queue depth grows; workers drain steadily. The spike becomes a slope
Failure isolationworkers crash or the encoder is down → jobs wait safely; nothing user-facing breaks (22's lessons apply to workers, not users)
Independent scalingqueue deepening? add workers — a scaling signal so clean that autoscalers (33) key off it directly

The mechanics that make it production-grade

  • Ack after work, not before: a worker takes a job (invisible to others, not deleted), does the work, then acknowledges. Crash mid-job → visibility timeout expires → job reappears for another worker. Nothing is lost — but note: this means jobs can be delivered twice (crash after work, before ack — case (c) again!). Workers must be idempotent. This is why lesson 22 came first.
  • Dead-letter queue (DLQ): a job that fails N times (poison message — malformed video, bug-triggering input) moves to a side queue for humans, instead of retrying forever and jamming the line.
  • Backpressure: producers outpacing consumers forever means unbounded queue growth — hiding the problem, then failing catastrophically. Bounded queues + "slow down" signals (429s upstream) surface the truth early. A queue absorbs bursts; it cannot absorb a permanent rate mismatch.

Hands-on: a queue in 30 lines (15 min)

Simulate the whole architecture with Python's stdlib — run it, then kill a worker mid-job and watch redelivery:

import threading, queue, time, random

jobs: queue.Queue = queue.Queue()

def producer():
    for i in range(20):                      # a burst of 20 uploads at once
        jobs.put({"job_id": i, "video": f"cat_{i}.mp4"})
    print("producer: burst enqueued, returning 202s to users")

def worker(name: str):
    while True:
        job = jobs.get()                     # take (blocks when empty)
        try:
            time.sleep(random.uniform(0.3, 0.8))          # "transcoding"
            if random.random() < 0.15:                    # simulated crash
                raise RuntimeError("worker died mid-job")
            print(f"{name}: done {job['video']}")
            jobs.task_done()                 # ack AFTER work
        except RuntimeError:
            print(f"{name}: CRASHED on {job['video']} — requeueing")
            jobs.put(job)                    # redelivery (real queues do this via timeout)
            jobs.task_done()

threading.Thread(target=producer).start()
for n in ("w1", "w2", "w3"):
    threading.Thread(target=worker, args=(n,), daemon=True).start()
jobs.join()
print("queue drained.")

Watch: the burst enqueues instantly (users are gone, happy); three workers drain it steadily; crashed jobs reappear and complete on another worker. Re-run with one worker vs five — that is independent scaling.

AI-lab tie-in

Batch inference is this lesson at GPU prices: requests queue up, GPU workers drain them in batches, queue depth drives autoscaling, poison prompts go to DLQs. And long-running agents ARE workers — a task queue feeding tool-wielding consumers, with idempotent steps because redelivery happens. The Anthropic Batches API's "submit now, results later" is a 202 + queue, productized.
