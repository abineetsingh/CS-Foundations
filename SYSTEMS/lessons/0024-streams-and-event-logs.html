Streams & Event Logs

One skill, learned today: distinguish the queue from the log — and know when "many readers, replayable history" means you want Kafka's shape, not SQS's.

Why this matters

"Queue or stream?" is a real architectural fork, and Kafka appears in virtually every large company's diagram. It's also the machinery behind a phrase you've already used twice — "fed asynchronously from the database" (search 0020, caches 0010). Today that arrow gets its proper name: the event log.

The log: a queue that remembers

A queue (0023) hands each job to one worker and deletes it on ack — the message's life is the work. An event log (Kafka, Kinesis, Redpanda) is different machinery: an append-only, ordered, durable sequence of events that consumers read without consuming. Each consumer just remembers its own position (offset) — like bookmarks in a shared book that nobody tears pages from [Kafka docs]:

Message queue (0023)Event log (today)
A message is…a job: "do this work"a fact: "this happened" (order_placed, tweet_posted)
Consumerscompete — each job to exactly one workerindependent — every group reads everything, at its own pace
After readingdeletedretained (days, or forever) — replayable
Reach for it whenwork distribution: transcode, email, thumbnailsmany systems need the same facts; history matters; audit; analytics

Scaling note that reuses your phase-2 brain: a Kafka topic is split into partitions by key — sharding (0016) applied to the log. Ordering is guaranteed within a partition only; choose the partition key (user_id, conversation_id) so the events whose order matters share one.

Why "facts, replayable, many readers" changes architecture

  • One write, N projections: tweet_posted is appended once; the search indexer, feed builder, notification sender, and analytics loader each consume it independently. Adding consumer #5 next year requires touching nothing upstream — and it can replay history from offset 0 to bootstrap. This is event-driven architecture.
  • CDC (change data capture): tail the database's own change log into the event log — now derived systems (search 0020, caches) are fed from truth automatically, not by app code remembering to tell everyone. The database stays authoritative; the log distributes its history.
  • Replay is a superpower with a bill: bugs in a consumer? Fix, rewind offset, reprocess — but consumers will see events again, so they must be idempotent (0022, always 0022).
AI-lab tie-in

Training-data pipelines are event-log-shaped end to end: user interactions append to streams; consumers fan them into feature stores, eval sets, and RLHF pipelines independently; replayability doubles as data lineage ("rebuild the dataset exactly as of March"). And an agent's action history is a private event log — append-only facts, replayable to reconstruct state. The log abstraction is everywhere once you can see it.
