Capstone: Build It, Then Defend It

The mission's success criteria, cashed in: ship a production-shaped service with your own hands, then survive a mock-interview marathon about it and its cousins.

The finish line, verbatim

From MISSION.md: "Build and run a small production-shaped service in Python: API + database + cache + queue, containerized, with basic observability" and "whiteboard a standard system design question in ~45 minutes." This lesson is both, scheduled. Expect 2–4 sessions for the build, one for the marathon.

Part 1 — The build: "Pastebin-with-a-pulse" (2–4 sessions)

A paste-sharing service — chosen because it exercises every layer without any single one exploding in scope. You've built every piece in isolation; the capstone is the composition. Requirements, as a production ticket:

#RequirementProves (lessons)
1POST /pastes (auth'd, rate-limited), GET /{code} — counter→base62 codes, 404/409/429 handled properly02–04, 28, 41
2Postgres (or SQLite) as truth; Redis cache-aside on reads with TTL + delete-on-write; measure and log your hit ratio09, 10, 12–13
3View counting OFF the read path: events to a queue (Redis list is fine), an idempotent worker aggregates — kill the worker mid-job and prove nothing is lost or double-counted22–25
4docker compose up brings up API + DB + Redis + worker; API and worker as separate containers from one image32
5/metrics with request counts by status and p50/p95/p99; structured JSON logs; a written SLO ("99.9% of reads < 100 ms") and a load test (hey or a Python loop) proving or breaking it35
6A DESIGN.md: the diagram, the envelope math for a pretend 10 M DAU, every tradeoff stated in "X because Y, alternative costs Z" form — written like a document a team would review06, 40, everything

Stretch goals (each one session): SSE endpoint streaming live view-counts (29) · a second API container behind an nginx load balancer, proving statelessness by killing one under load (07–08) · chaos drill: kill Redis under load, watch the DB take the storm, write the postmortem (36, 39).

Work with your teacher as a colleague on this — design reviews, debugging, code review — but type it yourself. The mission said build; typing is where the knowledge sets.

Part 2 — The mock marathon (1 session)

Ask your teacher to run, on separate days: (1) a full 45-minute mock on a question you haven't drilled (job scheduler, Ticketmaster, Dropbox — with real pushback and a written scorecard against the 40 stages); (2) a 45-minute defense of your capstone — "why this queue?", "what breaks at 100×?", "your cache dies — walk me through the incident" (defending real decisions is the closest simulation of senior loops that exists); (3) an AI-systems round from 44–45 territory. Score honestly against the runbook; a miss found here costs nothing.

After the capstone

  • Record it: have your teacher write learning records for what the build exposed — they steer any future deepening (Gossip Glomers, DDIA cover-to-cover, multi-region design).
  • Keep two habits: one postmortem per week (39's ritual — the danluu list is deep), and one drill re-run per month with fresh prompts, spaced (your DSA course taught you why spacing is the whole game).
  • Ship it somewhere real (Fly.io/Railway-class host — check RESOURCES.md's gap note): a deployed URL turns the capstone from exercise into evidence. Recruiters click links.
Course complete

Forty-six lessons ago, a server was a mystery box. Now you can design one, size it, scale it, break it, ship it, watch it, and defend every decision out loud — and the AI-systems frontier reads as familiar architecture rather than magic. The interview is a checkpoint; this capability was the actual goal. Go take both.
