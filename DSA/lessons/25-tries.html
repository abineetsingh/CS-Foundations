Tries: Trees of Prefixes

Today's skill: build a trie from scratch and know exactly when it beats a hash set — prefix queries.

Why this, now

The trie is a specialist: for exact lookup it loses to the hash map, but for "anything starting with…" it's unbeatable. Knowing precisely where each structure wins is the judgment interviews test. Bonus relevance: tokenizers — the front door of every LLM — lean on exactly this structure.

The structure: paths spell words

Each node's children are letters; a path from the root spells a prefix. Words sharing prefixes share paths — car, cart, and cat overlap for free:

root ── c ── a ─┬─ r (✓) ── t (✓)      (✓) = "a word ends here"
                └─ t (✓)
words: car, cart, cat — 4 nodes for 10 letters

Two ingredients per node: children (a dict — Lesson 05) and an end-of-word flag. Note what the flag does: car is both a word AND a prefix of cart — nodes alone can't distinguish "word" from "waypoint."

The implementation (worth memorizing — it's tiny)

class TrieNode:
    def __init__(self):
        self.children = {}          # char → TrieNode
        self.is_word = False

class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word):                    # O(L), L = len(word)
        node = self.root
        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()
            node = node.children[ch]           # walk/create the path
        node.is_word = True

    def search(self, word):                    # O(L) — exact word
        node = self._walk(word)
        return node is not None and node.is_word

    def starts_with(self, prefix):             # O(L) — the superpower
        return self._walk(prefix) is not None

    def _walk(self, s):
        node = self.root
        for ch in s:
            if ch not in node.children:
                return None
            node = node.children[ch]
        return node

Every operation is O(L) — the word's length — independent of how many words are stored. A dictionary of a million words: still L steps.

Trie vs. hash set: the honest comparison

OperationHash setTrie
exact lookupO(L)* — simpler, faster constantsO(L)
"any word with prefix p?"O(n·L) — scan everythingO(L) — walk p, done
list all words with prefix pO(n·L)walk p + DFS below (output-sized)
wildcard matching (c_t)poorDFS branching at the wildcard
spacecompactheavy (a node per char) — but shared prefixes compress

*hashing a string reads all L characters — hash lookups on strings were never truly O(1).

Decision rule: the word "prefix" (or autocomplete, or wildcard) in a problem statement → trie. Plain membership → hash set, always. The classic combo problem, "Word Search II" (find dictionary words in a letter grid), uses a trie to prune a grid DFS — Lessons 25 + 27 + 29 stacked; it will be within reach in three lessons.

Mission link · ML/AI

Fast BPE tokenizers store their merge vocabulary so that the longest matching token at each text position is found by walking a trie — that's how "unbelievable" becomes un + believ + able in microseconds. Constrained decoding (forcing an LLM's output to match a grammar or JSON schema) walks an automaton/trie of valid continuations while masking logits — a trie deciding what the model may say next.
