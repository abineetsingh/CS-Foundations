Delivery Guarantees

One skill, learned today: state which delivery guarantee a channel gives, why "exactly-once delivery" is a myth, and how "effectively-once" is really achieved.

Why this matters

"Does your queue deliver exactly once?" is a trap interviewers set deliberately — the confident wrong answer ("yes, we use Kafka") fails candidates who'd otherwise pass. The correct three-sentence answer is today's entire lesson, and it welds together everything from 22–24.

The three promises

Any channel moving messages between machines makes exactly one of these promises — forced by the (a)/(b)/(c) ambiguity of lesson 22 (no response ≠ not delivered):

GuaranteeMechanismFailure shapeAcceptable for
At-most-oncefire and forget — no ack, no retrymessages silently lostmetrics ticks, presence pings — losses vanish in aggregate
At-least-onceack + retry until confirmed (22's trio; 23's ack-after-work)messages duplicatedalmost everything — if consumers handle duplicates
"Exactly-once"as a delivery promise over a network: impossible. The sender of an unacked message must choose — resend (dupe risk) or not (loss risk). There is no third option [DDIA ch. 9]

Effectively-once: the real thing behind the marketing

What systems actually build is at-least-once delivery + idempotent processing = effectively-once effects. The duplicate arrives; its impact doesn't. The standard dedupe toolkit:

  • Idempotency keys / event ids (22): processed-id set; repeats are recognized and replayed, not re-executed.
  • Natural idempotency: make the operation an upsert — SET status='shipped' twice is harmless; counter += 1 twice is not. Prefer absolute writes over relative ones where possible.
  • Transactional dedupe: record the event-id in the same database transaction (14) as the effect — so "did the work" and "remembered doing it" can never disagree. This little move previews lesson 27's outbox pattern.

The interview-ready summary, three sentences: "Exactly-once delivery is impossible over an unreliable network — an unacknowledged send forces a choice between possible loss and possible duplication. So we choose at-least-once and make processing idempotent, deduping by message id. Kafka's 'exactly-once semantics' is this same recipe packaged — idempotent producers plus transactional consumers — not a physics exemption." [Kafka: message delivery semantics]

AI-lab tie-in

Usage metering at an LLM API is a delivery-guarantee decision with revenue attached: token-count events flow from inference servers to billing. At-most-once undercharges (lost events = free tokens); naive at-least-once overcharges (duplicate events = double-billed customers — trust-destroying). Real systems: at-least-once + event-id dedupe in the billing store. You now know why that design is forced, not chosen.
