Dynamic Programming IV: Knapsack

Today's skill: recognize "choose items under a budget" problems — 0/1 vs unbounded — and the state that solves them all.

Why this, now

Knapsack is DP's most-disguised family: "Partition Equal Subset Sum," "Coin Change," and "Target Sum" look unrelated until you see the costume — items, a capacity, take-or-skip decisions. Learn the two canonical forms and the disguises fall away.

0/1 knapsack: each item once

Items with weights and values; capacity W; maximize value. State: dp[i][w] = best value using the first i items within capacity w. Transition: item i's last-move menu — skip it, or (if it fits) take it:

def knapsack(weights, values, W):
    n = len(weights)
    dp = [[0] * (W + 1) for _ in range(n + 1)]
    for i in range(1, n + 1):
        for w in range(W + 1):
            dp[i][w] = dp[i-1][w]                          # skip item i
            if weights[i-1] <= w:                          # take item i:
                dp[i][w] = max(dp[i][w],
                               dp[i-1][w - weights[i-1]] + values[i-1])
    return dp[n][W]

O(n·W) — and pause on that: W is a number, not a count. This is called pseudo-polynomial: fine for W = 10,000, hopeless for W = 10¹⁸. Mentioning that distinction unprompted is a strong signal. Note the crucial index in the take-branch: dp[i-1][…] — the previous row — is what enforces "each item once."

Unbounded knapsack: items reusable — Coin Change

Fewest coins (reusable denominations) summing to amount. Reuse = the take-branch stays in the same row — which collapses naturally to 1-D:

def coin_change(coins, amount):
    dp = [float("inf")] * (amount + 1)     # dp[a] = fewest coins making a
    dp[0] = 0                              # zero coins make zero
    for a in range(1, amount + 1):
        for c in coins:
            if c <= a and dp[a - c] + 1 < dp[a]:
                dp[a] = dp[a - c] + 1      # last coin was c
    return dp[amount] if dp[amount] != float("inf") else -1

O(amount × len(coins)). The inf sentinel marks unreachable amounts — arithmetic-safe and checkable at the end (compare −1-style sentinels, which poison min).

The disguises

Problem as statedUndisguised
"Partition array into two equal-sum halves?"0/1 knapsack: can items fill capacity total/2 exactly?
"Fewest coins for amount" / "ways to make amount"unbounded knapsack (min / count variants)
"Assign +/− signs to hit a target"algebra first: positives must sum to (total+target)/2 → 0/1 count
"Can you form the string from dictionary words?" (Word Break)unbounded knapsack where 'capacity' is the string prefix

The recognition cue: a budget/target/capacity + items that consume it + take-or-skip choices. Then one question routes you: can items repeat? No → 0/1 (previous row). Yes → unbounded (same row). That index difference is the entire fork between the two forms — respect it, it's where knapsack bugs live.

Mission link · ML/AI

Resource allocation under budgets is ML-infrastructure daily life: packing training jobs into GPU-hours, selecting the best set of fine-tuning examples under a token budget, scheduling inference batches under a memory cap — knapsack relaxations everywhere. (True optimal is NP-hard, hence the pseudo-polynomial caveat and the greedy/approximate solutions production actually ships — Lesson 0038's territory.)
