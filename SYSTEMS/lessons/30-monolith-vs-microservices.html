Monolith vs Microservices

One skill, learned today: argue both sides of the architecture question every enterprise fights about — and know what a service boundary actually costs.

Why this matters

This is the "enterprise platforms" heart of your mission: how large organizations structure backends. Interviews pose it directly ("would you split this into services?") and indirectly (every box you draw implies an answer). The mature position is unfashionable and correct — and phase 3 gave you the vocabulary to hold it.

The two shapes

A monolith is one deployable unit: all features, one codebase, one process (× N stateless copies — a monolith scales horizontally just fine, per 07!). Microservices split the system into independently deployed services — orders, payments, inventory — each owning its own data, talking over the network [Fowler: microservices]:

A cross-feature call is…a function call: nanoseconds, can't fail alone, transactional (14 covers all of it!)a network call: milliseconds, needs 22's whole trio, no shared transactions (27's sagas)

Read the left column with phase-3 eyes: microservices import every hard problem from lessons 22–27 into the interior of your application. That's not a reason never to split — it's the price tag, and most of the industry's microservice regret comes from not reading it. The main legitimate driver is organizational: team autonomy (Conway's law — architecture mirrors org structure). The honest doctrine: monolith first; split when a specific pain (team contention, divergent scaling, isolation requirements) names the boundary [Fowler: Monolith First].

Drawing the boundary well

When you do split, two rules carry most of the weight:

  • Services own their data. No shared database — otherwise you've built a distributed monolith: network calls' costs, none of the autonomy (schema changes still need everyone's sign-off). Other services get your data through your API or your events (24), never your tables.
  • Split along business capabilities, not technical layers. "Payments" is a service; "the database access layer" is not. A good boundary is one where most changes stay inside it — cohesion over fashion.

The front door of a service fleet is the API gateway — one public address (clients shouldn't know your org chart) doing auth, rate limiting (28), and routing; internally, services find each other via service discovery (33) and increasingly talk through events rather than direct calls (24, 27) to keep coupling loose:

AI-lab tie-in

A production LLM platform is a service fleet you can now read: gateway (auth, token-based rate limits — 28) → routing service → inference fleet, with billing, safety, and logging as event consumers (24) so the request path stays fast. And multi-agent systems re-import the same question: one big agent or specialized agents coordinating over messages? The tradeoffs transfer verbatim — coordination cost vs specialization, again.
