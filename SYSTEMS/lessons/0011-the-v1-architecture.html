Why this matters

The assembly

Every box below earned its place in a previous lesson. Watch a single read request cross the whole thing, then close your eyes and redraw it — that retrieval is the actual exercise:

Two honest gaps in v1, flagged so you know the map isn't finished: the database is still a single box (replication and sharding are phase 2), and everything here is synchronous — no queues, no background work (phase 3). Interviewers respect hearing you name the limits of your own diagram.

Drill: Twitter-lite, for real this time

Paper and pen — the medium matters; interviews happen on whiteboards, not in editors. Work each step before opening its answer. Design: a text-only Twitter clone — post tweets, follow users, read a home timeline.

Step 1 — Requirements (3 minutes)

List functional requirements (what it does) and non-functional ones (how well). Aim for 3–4 of each.

Compare your answer

Functional: post a tweet (≤280 chars) · follow/unfollow · home timeline of followees' tweets, newest first · view a user's tweets. Non-functional: 10 M DAU scale · reads dominate ~100:1 · timeline should feel instant (<200 ms) · seconds-stale timelines acceptable — availability over freshness. That last sentence is a tradeoff declaration; interviewers reward it.

Step 2 — Envelope (3 minutes)

10 M DAU, 100 timeline reads/day each, 1 tweet/day each, tweet ≈ 0.5 KB. Compute read QPS, write QPS, storage/year. (Lesson 0006 method — no peeking at it.)

Compare your answer

Reads: 10⁹/day ÷ 10⁵ ≈ 10,000 QPS (peak ~50k). Writes: 10⁷ ÷ 10⁵ ≈ 100 QPS. Storage: 5 GB/day ≈ 1.8 TB/yr ×3 replicas. Conclusion to say out loud: writes trivial, storage modest, reads are the entire problem → cache hard.

Step 3 — API (3 minutes)

Write the endpoints, lesson-0002 style.

Compare your answer 
POST /tweets                     create (body: text)
GET  /timeline?limit=20&cursor=…  home timeline, paginated
POST /users/42/follow            follow user 42
DELETE /users/42/follow          unfollow
GET  /users/42/tweets?limit=20   a user's page

Nouns, pagination, and a cursor (not offset) for the timeline — infinite scroll over shifting data.

Step 4 — Draw the architecture (10 minutes)

Place: clients, DNS, CDN, LB, app servers, cache, database. Draw the read path and the write path as separate arrows. Annotate each box with one phrase saying why it's there, tied to the numbers from step 2.

Compare your answer

Client → DNS → CDN (avatars/static; API passes through) → LB (health-checked, least-connections fine) → stateless app fleet (scale with read load) → Redis (timelines + hot tweets, TTL ~60s + delete-on-write; target h ≥ 99% so the DB sees ≤100 read QPS) → DB (single primary handles 100 write QPS comfortably... for now). Write path: app → DB insert → delete affected cached timelines. Bonus points: noting the celebrity problem (1 tweet invalidates 10 M timelines) — real answer in lesson 0043.
