AI Supply Chain: Models, Data, and Code

One skill, learned today: establish provenance and safe loading for models, datasets, prompts, packages, and generated dependencies.

Mission connection

This skill supports both sides of the mission: explain the security decision clearly under interview pressure, then apply it to production software in a way that survives different clients, frameworks, and deployment stacks.

The security model

AI supply chains include executable model formats, training and evaluation data, adapters, prompts, libraries, registries, and generated code.

QuestionAnswer to make explicit
What must remain true?Write one observable invariant before choosing a tool or payload.
Where is it decided?Name the component holding authoritative identity, state, or key material.
What can the adversary vary?List reachable inputs, ordering, timing, identity, dependencies, and failure paths.
What proves the control?Keep legitimate behavior passing and adversarial behavior failing for the intended reason.

The failure pattern

Hallucinated packages are installed, pickle-like models execute on load, datasets are poisoned, public registries impersonate organizations, or mutable artifacts evade scanners.

Root-cause question

Do not ask only “which payload works?” Ask: which trusted interpretation, policy decision, state transition, or authority changed because of attacker-controlled data or timing? That answer transfers across Python, JavaScript, web, mobile-backed APIs, services, and cloud platforms.

Watch the decision cross its boundary

AI system attack path

Step through the full application, not only the model. Dashed nodes contain attacker-influenced content; purple nodes can create real-world consequences.

Design the durable control

Use approved registries, immutable digests, signatures and provenance, safe formats, isolated inspection, dataset lineage and sanitization, dependency review, and behavioral baselines.

Prefer controls that make the safe path easy and centralized. Prevention addresses the root transition; containment limits blast radius if prevention fails; detection makes unexpected use visible. These layers should fail independently rather than repeat the same assumption three times.

Production implementation shape

Promote immutable artifacts

artifact = registry.resolve("model@sha256:<digest>")
verify_signature(artifact, trusted_signers)
verify_manifest(artifact, approved_license_and_format_policy)
with isolated_no_network_loader() as loader:
    inspected = loader.inspect_without_executable_deserialization(artifact)
behavioral_gate.compare(inspected, approved_baseline)

Scanning complements—but does not replace—identity, provenance, safe formats, isolated loading, and behavioral evaluation.

Verify instead of believing

Rebuild from declared inputs, verify digest and signer, load artifacts without code execution, plant poisoned rows, compare behavior, and rehearse revocation and rollback.

Practice the complete loop

Authorized AI lab · deterministic sandbox

Scope: only the synthetic scenario and browser widget on this page, plus files you create specifically for it. Use fake tenants, canary secrets, documents, identities, and tool effects. Prohibited: real provider credentials, third-party agents, live inboxes, production vector stores, and external tool targets. Stop: when the synthetic security outcome and its policy decision are observable.

  1. Write the user-authorized goal, protected asset, untrusted content sources, and permitted effects.
  2. Run the legitimate case and record the expected model proposal and deterministic policy result.
  3. Introduce one synthetic adversarial variable—prompt, retrieved content, tool argument, artifact, credential, or quota—and predict the violated invariant.
  4. Apply the lesson control outside the model where enforcement is required; rerun allowed, denied, and failure cases.
  5. Preserve only sanitized test inputs, decision traces, metrics, and the regression. Delete transient prompts and synthetic secrets when finished.
