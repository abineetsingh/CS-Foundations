Lesson 05 · ~20 minutes · Systems Design

Latency Numbers Every Architect Knows

One skill, learned today: estimate how long a request takes by counting its hops — using a dozen numbers you'll keep for life.

Why this matters

Architects disagree about many things, but never about physics. These rough numbers — popularized by Jeff Dean of Google — are the shared arithmetic behind every design tradeoff you'll ever argue: why caches exist, why round trips are the enemy, why "just call the database in a loop" melts systems. Interviewers expect you to reach for them unprompted.

Three anchors worth saying out loud: memory is nanoseconds, SSDs are microseconds, networks are milliseconds. Each tier is roughly a thousand times slower than the one above. A cross-continent round trip is ~a million times slower than a RAM read — that ratio, scaled to human time, is "1 second vs 11 days."

The skill: count the round trips

A request's latency is dominated by its slowest hops — and hops add up when sequential. Estimate by walking the path:

Client (San Francisco) → API (Virginia datacenter), then 3 DB queries run one-by-one:

  cross-country round trip           ~70 ms
  DB query 1 (same datacenter)       ~0.5 ms + ~1 ms work ≈ 1.5 ms
  DB query 2                          ≈ 1.5 ms
  DB query 3                          ≈ 1.5 ms
  ─────────────────────────────────────────────
  total                              ≈ 75 ms   ← the WAN hop is almost everything

Two design moves fall straight out of the arithmetic:

  • Cut round trips. 100 sequential DB queries at ~1.5 ms each = 150 ms of pure chatter — the classic N+1 query problem. One query returning 100 rows: ~2 ms. Batch, join, or fetch in parallel.
  • Move data closer. Can't shorten the speed of light — so put a copy near the reader. That's caching (lesson 09) and CDNs (10), justified entirely by this table.
AI-lab tie-in

The same hierarchy governs GPUs: on-chip SRAM is ~100× faster than the GPU's HBM memory, which dwarfs PCIe and network links to other GPUs. LLM inference is usually memory-bandwidth-bound — the KV-cache exists to avoid recomputing, and techniques like FlashAttention exist to avoid re-reading slow memory. Same physics, different box.
