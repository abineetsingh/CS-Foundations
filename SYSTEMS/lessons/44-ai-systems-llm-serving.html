Lesson 44 · ~30 minutes · Systems Design · Phase 5

AI Systems I: LLM Serving

One skill, learned today: design an LLM inference API — continuous batching, KV-cache economics, model routing — using the systems toolkit you already own.

Why this matters

This is the mission's home turf: AI-lab interviews increasingly ask "design ChatGPT's backend" or "serve a 70B model at scale" — and the punchline of this lesson is that you already know 80% of the answer. The GPU adds new physics; the architecture around it is lessons 01–43 wearing a lab coat.

The new physics, in three facts

  • Inference is memory-bound and stateful-ish. Generating each token requires streaming the model's weights through the GPU (the 05 tie-in: HBM bandwidth is the wall), and attention needs the KV-cache — per-conversation attention state that grows with context length, gigabytes for long chats. It's your 09 cache concept, except losing it means expensive recomputation of the whole prefix, not a cheap DB refill.
  • Requests are wildly heterogeneous. One request: 50 tokens in, 10 out (100 ms). Another: 100k in, 4,000 out (minutes). This is 08's "least connections" argument at its extreme — round robin is malpractice here.
  • The unit of efficiency is the batch. A GPU serving one request at a time idles its compute; continuous batching (vLLM's contribution) admits and retires requests per token step, keeping the batch full as requests finish — 2–10× throughput over naive batching [vLLM docs].

The genuinely new design tensions worth naming in interviews: latency vs throughput is a dial (bigger batches = better GPU economics = worse time-to-first-token; interactive and batch traffic want different dial settings — hence separate pools); prefill vs decode (ingesting the prompt is compute-heavy, generating is memory-heavy — big shops disaggregate them onto different GPU pools); and prompt caching as product (shared system-prompt prefixes = KV-cache reuse across requests = the 10× cheaper cached-input pricing you've seen on real price sheets — 09 hit-ratio economics, billed to customers).

Interview framing

Asked to "design an LLM API," run the 40 algorithm unchanged — requirements (latency tiers! streaming!), envelope (tokens/sec, not QPS: 10k concurrent users × 20 tok/s each = 200k tok/s ÷ ~2k tok/s per GPU ≈ 100 GPUs before batching efficiencies), API (POST /v1/messages, SSE), HLD (the diagram above), deep dives (batching, KV-affinity routing, priority shedding). The examiner is watching whether GPU exotica breaks your process. It shouldn't — it's capacity math with pricier boxes.
