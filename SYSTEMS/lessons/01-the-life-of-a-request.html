Lesson 01 · ~25 minutes · Systems Design

The Life of a Request

One skill, learned today: trace a single request from a client to a server and back, naming every stop on the way.

Why this, first

Every system design interview answer — URL shortener, chat app, news feed — is drawn on top of one picture: a client sends a request, a server answers it. Every production backend you'll ever build, including agentic systems calling LLM APIs, is an elaboration of that same picture. Today you learn the picture itself, precisely.

Two roles, not two machines

The client is whoever asks: a browser, a phone app, or a Python script. The server is a program that sits waiting, listening for requests, and answers them.

This is the single most common beginner misconception, so let's kill it now: a server is not a special computer. It's software — often just a Python program — that listens on a network. The machine it runs on can be a laptop, a rack in a datacenter, or a rented cloud VM. One machine can run many server programs at once; they stay out of each other's way using ports — numbered doors, one per program. Web traffic conventionally knocks on port 443 (HTTPS) or 80 (HTTP) [MDN: How the web works].

To send anything, the client needs the machine's IP address — a numeric address like 203.0.113.7. But you typed api.example.com. Turning names into IP addresses is the job of DNS, the internet's phone book [MDN: How browsers work]. That's why DNS is always the first stop.

What travels on the wire: HTTP

Client and server agree on a message format called HTTP. A request says what to do and to what; a response says how it went and carries the result [MDN: HTTP overview]:

Request:               Response:
GET /users/42          HTTP/1.1 200 OK
Host: api.example.com  Content-Type: application/json
                       {"id": 42, "name": "Ada"}
PieceWhat it means
GETThe method: read something. (POST = create, PUT = update, DELETE = delete.)
/users/42The path — which endpoint, which thing.
200 OKThe status code. 2xx = success, 4xx = your request was wrong (404 = not found), 5xx = the server broke [MDN: status codes].
{"id": 42, …}The body — the actual data, almost always JSON for APIs.

Watch one request live its life

Here is GET https://api.example.com/users/42, end to end. The fourth box is the database — the program whose only job is keeping data safe and queryable, so the app servers themselves can stay disposable. Step through slowly; the goal is to be able to redraw this from memory.

Speed of things

Each arrow above costs time — latency. Rough feel: reading memory is ~100 nanoseconds, an SSD read ~100 microseconds, a round trip inside one datacenter ~half a millisecond, and a cross-country round trip ~40–70 milliseconds. Network hops dominate everything else — which is why so much of systems design is about avoiding them. (Full table in a later lesson.)

AI-lab tie-in

When your Python script calls the Anthropic API, this exact diagram runs: your script is the client, api.anthropic.com resolves via DNS, an app server receives POST /v1/messages, GPU-backed model servers play the role the database plays here, and a 200 with JSON comes back. Agent frameworks are this loop, repeated.

Hands-on: run a real server (5 minutes)

Python ships with a tiny web server. In one terminal, start it:

cd ~ && python3 -m http.server 8000

That program is now a server: listening on localhost (this machine, no DNS needed), port 8000. In a second terminal, be the client — and ask curl to show the raw conversation with -v:

curl -v http://localhost:8000/

Read the output: lines starting with > are the request your client sent (find the method and path); lines with < are the response (find 200 OK and Content-Type). Now request something that doesn't exist:

curl -v http://localhost:8000/nope

You get a 404 — and notice the first terminal logged both requests as they arrived. You have now run a server, sent raw HTTP, and read both halves of the conversation. Stop the server with Ctrl-C.

Check yourself

From memory — no scrolling up. Wrong answers teach more than right ones.
