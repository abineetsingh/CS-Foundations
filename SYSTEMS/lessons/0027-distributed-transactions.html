Distributed Transactions: Sagas & the Outbox

One skill, learned today: coordinate a multi-service operation without a shared BEGIN/COMMIT — sagas with compensation, and the outbox pattern that makes the events trustworthy.

Why this matters

Lesson 0014's warm blanket — atomicity — only covers one database. The moment "book flight + charge card + reserve hotel" spans services, there is no shared COMMIT. This is the signature problem of microservice and enterprise architectures, a standard interview deep-dive ("how do you keep them consistent?"), and the reason agent workflows are hard to make safe.

Why not just lock everything? (2PC and its price)

The classic answer, two-phase commit: a coordinator asks every participant "can you commit?" (phase 1, all lock their changes), then orders "commit!" (phase 2). It works — and production mostly avoids it, for one brutal reason: between phases, every participant holds locks and waits. If the coordinator dies, they wait… indefinitely, blocking other transactions across all services. 2PC couples the availability of everything to the flakiest participant — the exact coupling distributed design tries to break [DDIA ch. 9]. Know what it is; be ready to say why you'd avoid it.

The saga: a sequence with an undo script

A saga replaces the global transaction with local transactions in sequence, each with a compensating action — a prepared "undo" (refund, release, cancel). Success path: each step commits locally and triggers the next (via events — 0024). Failure at step k: run compensations for steps k−1…1, in reverse [microservices.io: saga]:

Two honest properties to state in interviews: the system is eventually consistent during the saga — there's a window where the card is charged but the hotel isn't booked (design reads to tolerate it, mark orders "pending"); and compensations are business undos, not database rollbacks — "refund" is a new action with its own failure modes (retry it with 0022's trio; it must be idempotent).

The outbox: making "commit + publish" atomic after all

Sagas run on events — which exposes a sneaky dual-write bug: the service commits to its DB, then publishes "order_created" to the log… and crashes between the two. Database says yes; downstream never hears. The fix is elegant enough to memorize forever — the outbox pattern:

BEGIN;
  INSERT INTO orders (…);                 -- the business write
  INSERT INTO outbox (event_type, payload)  -- the event, AS A ROW
    VALUES ('order_created', …);
COMMIT;                                    -- ← ONE atomic transaction (0014!)

-- separately: a relay tails the outbox table (or CDC — 0024)
-- and publishes each row to the event log, at-least-once.

The event becomes a row in the same local transaction — 0014's atomicity, smuggled into messaging. The relay might publish twice (crash before marking sent), and that's fine: consumers dedupe (0025). Chain complete: outbox → at-least-once → idempotent consumers → effectively-once sagas. That sentence is phase 3 in miniature.

AI-lab tie-in

An agent booking your trip IS a saga executor: each tool call is a local step (with 0022's timeout/retry/idempotency), and when step 3 fails, something must run the compensations for steps 1–2. Durable-workflow engines (Temporal and kin) — increasingly the backbone of production agent infra — are exactly saga machines: they persist every step so crashes resume instead of restart. Today's pattern is tomorrow's agent runtime.
