Sliding Window, Fixed Size

Today's skill: compute something over every length-k window of an array in O(n), not O(n·k) — by updating instead of recomputing.

Why this, now

Sliding window is two pointers' sibling: both pointers move the same direction, framing a contiguous chunk. "Every subarray of size k" questions are interview staples, and the incremental-update idea generalizes far beyond arrays.

The wasteful way

Find the maximum sum of any k consecutive elements.

def max_window_slow(arr, k):
    best = float("-inf")
    for i in range(len(arr) - k + 1):
        best = max(best, sum(arr[i:i+k]))   # O(k) work, n times
    return best

Each window is summed from scratch: O(n·k). But adjacent windows share k−1 elements — recomputing them is pure waste.

The window update

When the window slides one step right, exactly one element enters and one leaves:

def max_window(arr, k):
    window = sum(arr[:k])            # first window: O(k), once
    best = window
    for i in range(k, len(arr)):
        window += arr[i]             # new element enters
        window -= arr[i - k]         # old element leaves
        best = max(best, window)
    return best

O(n) time, O(1) space. The general recipe: keep a running summary of the window (sum, count-dict, max-so-far) that can be updated in O(1) when one element enters and one leaves.

Note what does not slide in O(1): the median, or "is the window sorted" — some summaries can't absorb a departure cheaply. Recognizing which is which is the skill.

Mission link · ML/AI

A 1-D convolution is literally a fixed-size window sliding over a signal, and the same recompute-vs-update economics govern fast conv implementations. Sliding-window attention (Mistral, Longformer) restricts each token to attend to a fixed-size window — turning the O(n²) attention of Lesson 01 into O(n·k).
