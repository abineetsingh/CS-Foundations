Design Problems: Composing Structures to Hit Complexity Targets

Today's skill: the "design a class where every operation must be O(…)" genre — solved by composing two structures that cover each other's weaknesses.

Why this, now

Design questions are the closest LeetCode gets to real engineering, which is why AI labs favor them — LRU Cache is quite possibly the single most-asked interview question at infrastructure-heavy companies. The skill is new: not finding an algorithm, but composing data structures so every required operation hits its complexity budget.

The genre's one insight

No single structure does everything (Lesson 21's table said so). But requirements like "O(1) lookup AND O(1) ordering maintenance" decompose: use two structures, each covering one requirement, kept in sync. The design work is choosing the pair and defining the sync discipline.

Worked: LRU Cache (the boss question)

Fixed capacity; get(key) and put(key, value) both O(1); on overflow, evict the least-recently-used item.

Requirements → structures: O(1) key lookup screams hash map (05); "recency order with O(1) move-to-front and evict-from-back" screams doubly linked list (12, plus prev pointers). Neither alone suffices; wired together they do — the map's values are pointers to list nodes:

class Node:
    def __init__(self, key, val):
        self.key, self.val = key, val
        self.prev = self.next = None

class LRUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.map = {}                          # key → Node
        self.head = Node(0, 0)                 # dummy MRU end (12's armor, ×2)
        self.tail = Node(0, 0)                 # dummy LRU end
        self.head.next, self.tail.prev = self.tail, self.head

    def _unlink(self, node):
        node.prev.next, node.next.prev = node.next, node.prev

    def _push_front(self, node):               # mark as most-recent
        node.next, node.prev = self.head.next, self.head
        self.head.next.prev = node
        self.head.next = node

    def get(self, key):
        if key not in self.map:
            return -1
        node = self.map[key]
        self._unlink(node); self._push_front(node)     # touching = refreshing
        return node.val

    def put(self, key, value):
        if key in self.map:
            self._unlink(self.map[key])
        node = Node(key, value)
        self.map[key] = node
        self._push_front(node)
        if len(self.map) > self.cap:
            lru = self.tail.prev                # least-recent lives at the back
            self._unlink(lru)
            del self.map[lru.key]               # node stores key for THIS line

Every operation: a dict op + a few pointer rewires — all O(1). Three design details worth narrating in an interview: why doubly linked (unlinking needs the predecessor in O(1)); why nodes store keys (eviction must delete from the map); why two dummies (no head/tail special cases — Lesson 12's trick, doubled). In production you'd say OrderedDict or functools.lru_cache — name them, then build it anyway; the building is the question.

The same insight, twice more (recognition level)

  • Min Stack — O(1) push/pop/min: a stack, plus a parallel stack of running minimums. Two structures moving in lockstep; the second answers the query the first can't.
  • Insert/Delete/GetRandom O(1) — dict + list: dict maps value→index; random picks from the list; deletion swaps the target with the last element then pops (O(1) — the swap-with-last trick, worth knowing by name).

The interview procedure: write each required operation with its budget → pick a structure per requirement → define what "in sync" means → walk every operation proving both structures stay in sync and on budget. Sync discipline is where bugs live (e.g., forgetting the map-delete on eviction) — state it explicitly.

Mission link · ML/AI

LRU is not an exercise — it's THE cache policy: CPU caches, OS page tables, CDN edges, and yes, LLM inference: prefix/prompt caches evict least-recently-used entries, and vLLM's KV-cache block manager is an LRU-flavored allocator (Lesson 12's PagedAttention thread, completed). You will build or debug a cache in your first year at any infra-adjacent AI job. This lesson is that, miniaturized.
