Drill: Web Crawler & Notification System

Two classics, one sitting — both secretly the same question: queues, retries, rate limits, and fan-out, composed under failure.

Why this matters

The phase 3 milestone. Both prompts are interview staples because they force async thinking — there's no synchronous design that works. Paper, pen, voice. Attempt fully before each reveal; diff honestly after.

Drill A — Web crawler (20 min)

Design a crawler that fetches 1 B pages/month, extracts links, and feeds pages to a search indexer. Be polite to websites.

A1 — Envelope & shape (5 min)

Fetch QPS? What's the core loop? Which phase-3 boxes appear?

Compare your answer

10⁹ ÷ (2.6×10⁶ s) ≈ ~400 fetches/s sustained. Core loop: URL frontier (a queue, 23) → fetcher workers → parse → new links back into the frontier → page content to the indexer via an event log (24) so multiple consumers (indexer, archive, analytics) read independently. The crawler IS a queue-worker system whose workers feed the queue.

A2 — The hard parts (10 min)

Handle: politeness (don't hammer one site), duplicate URLs, fetch failures, and crawler crashes. Name the exact mechanism for each.

Compare your answer

Politeness = per-domain rate limiting (28 token buckets, keyed by domain) + partition the frontier by domain (16/24 thinking) so one domain's queue can't flood the fetch fleet; respect robots.txt. Duplicates: seen-URL set — at 10B+ URLs, a probabilistic structure (Bloom filter) or sharded KV; dedupe is idempotency (25) for fetch work. Failures: timeouts + capped retries with backoff/jitter (22); permanently failing URLs → DLQ (23) with a retry-after-days policy. Crashes: ack-after-work (23) so in-flight URLs redeliver; fetch is naturally idempotent (re-fetching a page is safe).

Drill B — Notification system (20 min)

Design notifications for a social app: likes, comments, follows → push/email/in-app, 100 M events/day, users can mute channels. No duplicate notifications.

B1 — Shape & API (5 min)

Compare your answer

100 M/day ≈ 1,000 events/s. Shape: producing services emit domain events (like_created…) to an event log (they should NOT call a notification API synchronously — 30's loose coupling); a notification service consumes, applies preferences/mutes, renders per channel, and enqueues delivery jobs per channel (queues — push/email have wildly different rates and failure modes, so separate queues + workers per channel). In-app unread counts: cached counters (09). API: GET /notifications?cursor=…, PUT /preferences.

B2 — The guarantees (10 min)

"No duplicates" + "don't lose them" — which delivery guarantee, enforced where? What about a dead push-provider? Batching ("Ada and 12 others liked…")?

Compare your answer

At-least-once end to end (25) + idempotency at the delivery edge: notification id = hash(user, event, channel); the delivery worker checks-and-records it transactionally before sending (25's dedupe recipe — "exactly-once" enforced at the last hop, where it matters). Dead provider (APNs/email down): the channel's queue absorbs (23), retries with backoff; if hard-down, DLQ + alert; other channels unaffected — failure isolated per queue. Batching: aggregate in a short window (30–60 s) keyed by (user, object) before rendering — a deliberate latency-for-quality trade you should declare out loud. Muted channels drop at the consumer, cheaply, before any queueing.

Self-score

  • Did a queue/log appear in your first sketch, unprompted? (If your design was synchronous, re-read 23 — that's the phase's core reflex.)
  • For every network edge: did you state its timeout/retry/idempotency story? (22 is a per-edge tax, not a chapter.)
  • Did you place the dedupe exactly where duplicates become user-visible?
  • Note your misses for your teacher — they calibrate what to review before phase 4.
