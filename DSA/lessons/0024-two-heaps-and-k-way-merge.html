Two Heaps & K-Way Merge

Today's skill: two named patterns built from heaps — tracking a running median, and merging many sorted streams.

Why this, now

These two patterns close Lesson 0007's oldest loose end (the median doesn't slide) and generalize Lesson 0018's merge from 2 lists to k streams. Both appear on the NeetCode 150's hard tier — and both are real infrastructure algorithms, not interview inventions.

Pattern 1: Two heaps — median from a stream

Numbers arrive one at a time; report the median after each. Sorting per query: O(n log n) each. The insight: the median only cares about the frontier between the lower and upper halves — so keep each half in a heap facing the frontier:

import heapq

class MedianFinder:
    def __init__(self):
        self.lo = []   # MAX-heap (negated): lower half — its root is the frontier
        self.hi = []   # MIN-heap: upper half — its root is the frontier

    def add(self, num):                          # O(log n)
        heapq.heappush(self.lo, -num)            # 1. in through the lower half
        heapq.heappush(self.hi, -heapq.heappop(self.lo))   # 2. lo's best → hi (ordering)
        if len(self.hi) > len(self.lo):          # 3. rebalance sizes (lo may hold +1)
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def median(self):                            # O(1)
        if len(self.lo) > len(self.hi):
            return -self.lo[0]
        return (-self.lo[0] + self.hi[0]) / 2

Two invariants, maintained by the three-step dance: every element of lo ≤ every element of hi (step 2 forces each arrival through the frontier), and sizes differ by ≤ 1 (step 3). Given both, the median sits at the roots — O(1) to read, O(log n) to maintain. The tell: "median/middle/balance-point of a growing or sliding collection."

Pattern 2: K-way merge

Merge k sorted lists into one. Lesson 0018's two-pointer merge doesn't scale — k pointers means k comparisons per step: O(N·k). The heap fix: the next output is always the minimum among the k current fronts, and "min of a changing candidate set" is a heap's one job:

def merge_k(lists):                       # N total elements, k lists
    h = []
    for i, lst in enumerate(lists):       # seed: each list's front
        if lst:
            heapq.heappush(h, (lst[0], i, 0))     # (value, which list, index)
    result = []
    while h:
        val, i, j = heapq.heappop(h)      # smallest front anywhere — O(log k)
        result.append(val)
        if j + 1 < len(lists[i]):         # that list's NEXT element replaces it
            heapq.heappush(h, (lists[i][j + 1], i, j + 1))
    return result

O(N log k) time, O(k) heap. The heap never holds more than one element per list — k candidates, not N. Tells: "merge k sorted anything," "smallest range covering one element from each list," "kth smallest in a sorted matrix" (each row is a stream).

Mission link · ML/AI

K-way merge is how big systems combine sharded results: a vector database retrieves top-k per shard, then k-way merges the score streams (external sorting in databases is exactly this too). The two-heap frontier idea — maintain just enough order at a boundary — is how streaming percentile monitors (p50/p99 latency dashboards) work at scale.
