Lesson 0026 · Phase 2 · ~30 minutes

Backtracking I: Choose, Explore, Un-choose

Today's skill: generate all subsets and permutations with one template — recursion that cleans up after itself.

Why this, now

Backtracking is systematic trial of every possibility — the honest algorithm when no clever shortcut exists. "Generate all…" questions are interview regulars, and the template here is also the skeleton of DFS-based agents exploring action spaces. It's recursion's final exam before Phase 3.

The mental model: walking a decision tree

Every "generate all X" problem is a tree of decisions. For subsets of [1, 2, 3]: each element asks in or out?

                     []
              in 1 ↙     ↘ out
              [1]           []
          in 2↙  ↘out    in 2↙  ↘out
         [1,2]   [1]     [2]     []
         ↙  ↘    ↙  ↘    ↙  ↘   ↙  ↘        8 leaves = 2³ subsets

Backtracking is DFS (Lesson 0019) on this implicit tree — it exists only in the call stack, one path at a time.

The template

def subsets(nums):
    result = []
    path = []                            # the current branch's choices

    def backtrack(i):                    # deciding about nums[i]
        if i == len(nums):               # every decision made: a leaf
            result.append(path[:])       # snapshot! (see below)
            return
        path.append(nums[i])             # CHOOSE:   nums[i] in
        backtrack(i + 1)                 # EXPLORE
        path.pop()                       # UN-CHOOSE: nums[i] out
        backtrack(i + 1)                 # EXPLORE the other branch

    backtrack(0)
    return result

The heartbeat is choose → explore → un-choose. The un-choose (path.pop()) restores path exactly as it was, so the sibling branch starts clean — one shared mutable list, safely reused across the whole tree, instead of copying a new list per call (which works, but costs a copy at every node instead of only at leaves).

Two bugs write themselves here; know both cold:

  • Missing pop → choices leak into sibling branches; results contain phantom elements.
  • result.append(path) without [:] → every "result" is the same list object, later emptied by pops. You return a list of ghosts. The snapshot copy is mandatory.

Permutations: same template, different choice set

def permutations(nums):
    result = []
    path = []
    used = [False] * len(nums)               # what's still available

    def backtrack():
        if len(path) == len(nums):
            result.append(path[:])
            return
        for i in range(len(nums)):           # every unused element is a branch
            if used[i]:
                continue
            path.append(nums[i]); used[i] = True     # choose
            backtrack()                              # explore
            path.pop(); used[i] = False              # un-choose
    backtrack()
    return result

Only the choice set changed: subsets ask in/out per element (2ⁿ leaves); permutations ask "which unused element next?" (n! leaves). Design any backtracker by answering three questions: what are the choices at each step? when is a path complete? what state must un-choose restore?

The cost, honestly

Output size is the floor: 2ⁿ subsets, n! permutations — exponential/factorial by nature (Lesson 0001's last row, arrived at last). n = 20 → a million subsets (fine); n = 40 → a trillion (never). State these numbers before coding; knowing when backtracking is hopeless is part of the skill. Making it less hopeless — pruning — is Lesson 0027.

Mission link · ML/AI

An agent trying tool-call sequences, a game engine trying move lines, a theorem prover trying tactics — all walk decision trees with choose/explore/un-choose (state restoration is why game engines "make/unmake" moves rather than copying boards). The reason LLM agents don't brute-force like this: 2ⁿ. The reason they work anyway: learned policies prune the tree — which is what beam search (Lesson 0023) and MCTS approximate.
