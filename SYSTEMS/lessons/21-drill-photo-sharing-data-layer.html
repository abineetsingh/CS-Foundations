Drill: Photo-Sharing Data Layer

One skill, practiced today: design a complete data layer — schema, stores, indexes, shards, replicas, consistency calls — for Instagram-lite, out loud.

Why this matters

This is the phase 2 milestone, run interview-style. The deep-dive half of a real system design round is exactly this exercise: "how is the data stored, found, scaled, and kept correct?" Paper and pen, answer before revealing, and — genuinely — speak your reasoning aloud. Verbalizing is the skill being trained.

The prompt

Design the data layer for a photo-sharing app: users post photos with captions, follow each other, view a home feed, and search captions. 50 M DAU. Each user views 50 photos/day and posts 0.2/day. Photo ≈ 2 MB; caption ≈ 200 bytes.

Step 1 — Envelope (5 min)

Read QPS, write QPS, photo storage/year, metadata storage/year. Which numbers dictate the design?

Compare your answer

Reads: 50M × 50 = 2.5×10⁹/day ÷ 10⁵ ≈ 25,000 QPS (peak ~100k+). Writes: 50M × 0.2 = 10⁷/day ≈ 100 QPS — trivial. Photos: 10⁷ × 2 MB = 20 TB/day ≈ 7 PB/year — dictates object storage + CDN, non-negotiable. Metadata: 10⁷ × ~1 KB ≈ 10 GB/day ≈ 3.6 TB/yr — one database's problem-but-manageable. Conclusion: read-heavy metadata + petabyte blobs: two different problems, two different stores.

Step 2 — Schema & stores (10 min)

Tables + which store each dataset lives in, with the reason (lesson 17's framework).

Compare your answer 
users  (id PK, name, bio)                        → Postgres
photos (id PK, user_id FK, object_key, caption,
        created_at)                              → Postgres (metadata only!)
follows (follower_id, followee_id,
         PK(follower_id, followee_id))           → Postgres
bytes: photos/{user}/{photo}.jpg                 → object storage (19)
captions, searchable                             → search cluster, fed async (20)
hot feeds + hot photo metadata                   → Redis, cache-aside (09)

Relational for the truth: real relationships (follows), constraints, and unanticipated queries. The blobs and search index are derived/specialized — each store earns its place with a named reason.

Step 3 — Indexes (5 min)

The two hot queries: a user's photos (newest first) and a feed built from followees' recent photos. Write the exact indexes.

Compare your answer 
CREATE INDEX idx_photos_user_time ON photos (user_id, created_at DESC);
-- follows: PK(follower_id, followee_id) already serves "who do I follow"
CREATE INDEX idx_follows_followee ON follows (followee_id);   -- "who follows X" (fan counts)

The composite (user_id, created_at) is the star: finds a followee's photos already time-ordered — filter column first, then order column (leftmost-prefix rule, 13). Verify with EXPLAIN, mention the write tax.

Step 4 — Scale & consistency calls (10 min)

25k read QPS, 3.6 TB/yr growth: replicas or shards — which, when, keyed how? Then assign consistency models: feed reads, "my profile after posting," follow counts, search.

Compare your answer

Scale, in order of reach (cheapest first): cache feeds/hot metadata at h≥99% → DB sees ~250 read QPS; add 2–3 read replicas for misses + headroom (15). Sharding not yet — 3.6 TB/yr fits one primary for years; say "shard by user_id via consistent hashing when the primary nears limits" (16) and note the celebrity hot-key caveat with its mitigations. Deferring correctly is senior-signal.

Consistency (18): feeds — eventual (seconds stale, invisible); own profile after post — read-your-writes (route poster's reads to leader briefly, 15); follow counts — eventual, cached; search — eventual, lag declared. Nothing here needs linearizable — saying that explicitly is the flex.

Self-score (honest!)

  • Did the envelope numbers change your design (object storage forced, sharding deferred)? That's the point of computing them.
  • Did every store have a spoken reason? Every index match a named query shape?
  • Did you declare staleness budgets per read path, unprompted?
  • Record what you missed as a note for your teacher — misses here define what to review before phase 3.
