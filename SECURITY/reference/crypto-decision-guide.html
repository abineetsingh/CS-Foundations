Practical Cryptography Decision Guide

Choose the property first, then a maintained high-level construction and key lifecycle.

NeedUseDo not substitute
Unpredictable tokenPython secrets or Web Crypto CSPRNGrandom, timestamps, UUID assumptions
Password verificationArgon2id; scrypt/PBKDF2 where constraints requireSHA-256, reversible encryption, homemade salting
Integrity/authenticity with shared keyHMAC through maintained APIBare hash, encrypted checksum, ordinary comparison
Confidential recordAEAD such as AES-GCM or ChaCha20-Poly1305ECB, unauthenticated CBC/CTR, custom composition
Publicly verifiable originModern signature API such as Ed25519MAC, “encrypt with private key,” custom RSA
Network channelModern TLS with hostname and chain verificationApplication-layer homegrown encryption

Five questions before code

  1. Which asset and attacker are in the threat model?
  2. Which property is required: secrecy, integrity, authenticity, freshness, or password resistance?
  3. Who owns and may use each key?
  4. How are nonces, versions, rotation, revocation, and failures handled?
  5. Which tamper, wrong-key, replay, and migration tests prove the claim?
Course rule

Do not design a primitive or combine low-level primitives. Prefer maintained high-level APIs; cryptography can be correct while key distribution, storage, or lifecycle still makes the system insecure.
