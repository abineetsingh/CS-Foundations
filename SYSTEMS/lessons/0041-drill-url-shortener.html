Drill: URL Shortener

The canonical opener, executed full-length under the 0040 algorithm — with a timer. It looks easy; the deep dives are where it bites.

Why this matters

TinyURL/bit.ly is the most-asked warm-up in the genre — interviewers use it to watch your process at full speed. Run it exactly as an interview: 45 minutes on a real timer, paper, speaking aloud, stages in order. Reveals only after each stage attempt.

The prompt

Design a URL shortener: long URL in, short link out; visiting the short link redirects. 100 M new links/month, 10 B redirects/month. Links live for years. Optional: custom aliases, click counts.

Stage 1+2 — Requirements & envelope (10 min)

Compare your answer

Functional: shorten(long)→short; redirect(short)→301/302; optional custom alias; click counts (descope live analytics — say so). Non-functional: redirects must be FAST (<50ms — they're in every user's critical path) and highly available (a dead shortener breaks the internet's links); read:write ≈ 100:1; eventual consistency fine except a fresh link must work immediately (read-your-writes for the creator — 0015/0018). Envelope: writes 10⁸/2.6M s ≈ 40/s; reads ≈ 4,000/s (peak ~20k). Storage: 10⁸ × ~500 B × 12 mo × 5 yr ≈ 3 TB — one primary + replicas territory, sharding deferred with a named trigger (0016, 0021's discipline). Verdict: tiny writes, hot reads, cache everything.

Stage 3 — API & data model (5 min)

Compare your answer 
POST /links {url, custom_alias?}   → 201 {short}     (auth'd, rate-limited — 0028!)
GET  /{code}                       → 301/302 Location: long_url

links (code TEXT PK, long_url TEXT, owner_id, created_at, expires_at?)

One real decision hides here: 301 (permanent) vs 302 (temporary). 301 lets browsers/CDNs cache the redirect — faster, less load (0010!) — but then you never see repeat clicks (bye, analytics) and can't update the target. Choosing 302 because analytics was a requirement is exactly the "X because Y, alternative costs Z" form (0040).

Stage 4 — High-level design (10 min)

Compare your answer

The 0011 shape, read-optimized: DNS → LB → stateless fleet → Redis cache-aside on code→url (hot links are wildly skewed — h≥99% plausible; 4k QPS becomes ~40 DB QPS) → Postgres primary + 2 replicas. Click counts: don't write on the redirect path — emit a click event to a queue/log (0023/0024); a consumer aggregates counts async. The redirect path touches cache only, nearly always.

Stage 5 — Deep dives (15 min)

Attempt each before opening: (a) how are codes generated — and why is "hash the URL and truncate" a trap? (b) custom aliases: what's the concurrency bug? (c) what breaks at 10× and what do you pre-position?

Compare: (a) code generation

Truncated hashes collide (birthday paradox — at 10⁸ links, 6-char collisions are certain) → collision-check-and-retry loops under concurrency. Cleaner: counter → base62 (7 chars covers 3.5 × 10¹²): collision-free by construction. The counter is a single point of contention → allocate ranges per app server (server A gets 1M ids at a time — a coordination fact for etcd/or a DB row, 0026's small-strong-core) → no per-request coordination. Mention the sequential-guessability nit (enumerate others' links) → append a couple random chars or bijective-scramble the counter. That whole paragraph IS the interview.

Compare: (b) custom aliases

Two users claim /claude simultaneously: check-then-insert is the lost-update shape (0014). Fix: let the PK do it — INSERT and catch the unique-constraint violation → 409 (0002!). The database is the arbiter; app-level checks are just UX prefetch.

Compare: (c) 10× and failure modes

Reads 10× → still cache-shaped (add replicas/nodes). Storage 10× → 30 TB → now shard by hash(code) (code lookups are always by exact key — perfect hash-shard fit, no scatter-gather, 0016). Redis down → 4k QPS hits Postgres: survivable IF you pre-warmed replicas — say "cache outage is my scariest failure; replicas are sized for cache-miss storms" (0036 thinking). Bonus flex: redirects at the edge (CDN workers reading a replicated KV) for global <20ms.
