Containers

One skill, learned today: package your API as a container image — the same artifact running identically on your laptop and in production — and understand what a container actually is.

Why this matters

Phase 4 is about running what you've designed. The container is production's unit of shipping: every deploy, every scaling event, every Kubernetes pod (0033) moves one. Interviews assume the vocabulary; the capstone requires the skill. And your notes API is about to become portable.

What a container actually is (and isn't)

The problem it kills: "works on my machine." Your API needs Python 3.12, FastAPI, system libraries, env vars — and production has different versions of everything. A container packages your code plus its entire userspace (interpreter, libraries, OS files) into an image that runs the same everywhere [Docker overview].

Crucially, a container is not a virtual machine. A VM virtualizes hardware and boots a whole guest OS (minutes, GBs). A container is just a normal process on the host, isolated by two kernel features: namespaces (it sees its own filesystem, network, process list) and cgroups (it's capped to its share of CPU/memory). Starts in milliseconds, costs ~nothing extra. One machine happily runs hundreds — which is what makes the cattle model (0007) economical.

TermIs
Imagethe frozen artifact: filesystem + metadata, built in layers, immutable, versioned (myapi:v42)
Containera running instance of an image (image : container :: class : instance)
Registryobject storage for images (Docker Hub, ECR) — push once, pull anywhere
Layerseach Dockerfile step stacks a filesystem diff; unchanged layers are cached and shared — why rebuilds and pulls are fast if you order steps well

Hands-on: containerize the notes API (15 min)

Install Docker Desktop (or the lighter OrbStack) once. Then in ~/notes-api, create Dockerfile:

FROM python:3.12-slim              # base layer: OS files + Python
WORKDIR /app
COPY requirements.txt .            # deps first — this layer caches…
RUN pip install -r requirements.txt
COPY main.py .                     # …while code (changes often) comes last
EXPOSE 8000
CMD ["fastapi", "run", "main.py", "--port", "8000"]

(Create requirements.txt containing fastapi[standard].) Build and run:

docker build -t notes-api:v1 .
docker run -p 8000:8000 notes-api:v1
curl localhost:8000/notes          # same API — now running inside the container

Poke at what you made: docker ps (it's a process), docker exec -it <id> sh then ls / (its own private filesystem — your Mac's files aren't there). The -p 8000:8000 maps host port → container port (0001's ports, now two layers deep). Now the payoff experiment: kill it, run it again — the notes are gone. Container filesystems are ephemeral: cattle, not pets (0007). Real persistence = mount a volume (-v) or, properly, use an external database — exactly the state/stateless split from 0004/0007, now enforced by the packaging.

Change main.py and rebuild: only the last layers rebuild (cache!), and the result is notes-api:v2 — a new immutable artifact. Deploys ship v2 alongside v1 and shift traffic — never "update" a running box. That's lesson 0034's whole foundation.

AI-lab tie-in

ML infra runs on this: training jobs ship as images (CUDA + frameworks + code — the "works on my GPU machine" problem is worse, and containers tame it), and agent sandboxes are containers wearing a security hat — namespaces/cgroups repurposed so untrusted agent code sees only its own little world. The isolation you just poked at with ls / is the same wall that keeps an agent's rm -rf inside the box.
