Greedy: When Local Best Is Globally Best

Today's skill: recognize when taking the locally best option is provably optimal — and when it's a trap that DP must clean up.

Why this, now

Greedy sits opposite DP: no table, no look-back — one decisive pass. When it's right it's beautifully simple; when it's wrong it's confidently, silently wrong. Interviews test exactly this judgment: candidates who greedy correctly and say why stand out from those who greedy hopefully.

The idea, and the trap, in one problem

Coin Change with coins [1, 5, 10, 25], amount 30: greedy (take the biggest coin that fits) gives 25+5 = 2 coins. Optimal! Same greedy with coins [1, 15, 25], amount 30: greedy gives 25+1+1+1+1+1 = 6 coins; optimal is 15+15 = 2. Identical algorithm, one works, one fails — greediness is a property of the problem, not the technique. US coin denominations happen to have the right structure; [1, 15, 25] doesn't. That's why Lesson 0037 solved general coin change with DP.

Two greedy classics that ARE correct

Jump Game — track the reachable frontier

nums[i] = max jump length from i; can you reach the end?

def can_jump(nums):
    reach = 0                           # furthest index reachable so far
    for i, jump in enumerate(nums):
        if i > reach:                   # standing beyond the frontier: stuck
            return False
        reach = max(reach, i + jump)    # extend the frontier
    return True

O(n), one pass, one variable. Why safe? Reachability is monotone — knowing the furthest reachable point summarizes everything useful about the past; no combination of shorter jumps can beat the frontier. (Notice the family resemblance to Lesson 0008's window invariant: one maintained quantity carries the whole history.)

Interval scheduling — earliest finish first

Maximum number of non-overlapping meetings. Sort by end time, always take the earliest-ending compatible meeting. The proof is the standard exchange argument, worth owning as a template: take any optimal schedule; its first meeting can be swapped for the earliest-ending one without breaking anything (it ends no later, so everything after still fits). Repeat down the schedule — greedy is never behind. Sorting by start time or by duration both fail (build the 3-interval counterexamples yourself — it's a five-minute exercise that cements the point).

The judgment procedure

  1. Try to break it: hunt for a small counterexample where local-best loses (like [1, 15, 25]). Two minutes of honest attack.
  2. If unbreakable, say why: an exchange argument ("any optimal solution can be rewritten to start with the greedy choice") or a monotone-frontier argument (Jump Game, Dijkstra's finalization — Lesson 0032).
  3. If broken, fall back to DP — the decision tree genuinely branches, so pay for the table (Lesson 0034's four questions).

In interviews, run the procedure out loud: "Greedy by X… can I break it? … here's a counterexample, so DP" is exactly the reasoning they're grading. Greedy tells: "maximum number of non-overlapping…", "minimum number of jumps/refuels/arrows", sorted-then-scan structure.

Mission link · ML/AI

Greedy decoding — take the argmax token every step — is the [1, 15, 25] trap at generation time: locally best tokens, globally mediocre sentences. Beam search (Lesson 0023) hedges by keeping k candidates; sampling hedges by randomizing. And gradient descent itself is a greedy frontier algorithm — locally steepest step, no global guarantee — which is why loss landscapes, momentum, and learning-rate schedules matter. "When is greedy safe?" is a research question your field asks daily.
